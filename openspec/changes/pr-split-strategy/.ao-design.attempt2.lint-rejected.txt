# Design: Detection Emulation Vertical Slice Merge Strategy (v2)

## Approach

The central problem is that `validate_rule_tool.ts` statically imports `scoreConfidence`, `createEmulationHistory`, `EmulationRunner`, and `DetectionEmulationGuardrails` at module load time (`validate_rule_tool.ts:35-43`). This creates a monolithic dependency graph that prevents sliced merging: PR 1a cannot build without every module that PRs 2, 3, and 4 introduce. The fix is an **optional enrichment pipeline** that converts those four hard dependencies into injectable, optional fields on a `ValidateRuleEnrichments` interface. The tool handler runs end-to-end at any level of enrichment, degrading gracefully when a capability has not yet been wired in. No callers outside the tool boundary are affected by this change.

The second layer of work is closing the three feature-flag registration gaps identified in the research: guardrails are constructed unconditionally in `plugin.ts` (`plugin.ts:659`), neither REST route is conditionally registered (`register_routes.ts:39-44`), and `detection_emulation_skill.ts` registers all six tools regardless of the `realExecution` flag (`detection_emulation_skill.ts:37-157`). Each gap is closed in the PR where the corresponding capability is introduced, so at every merge point the shipped code is inert by default.

The dependency ordering (1a → 2 → 3 → 4, with 1b merging independently into 3) enforces a clean capability ladder. Each PR adds enrichments and un-gates exactly the surface area it introduces. PRs never reach backward to unlock capability they do not own.

## Components

- **`ValidateRuleEnrichments` interface** (`validate_rule_tool.ts`): Declares `scoreConfidence`, `createEmulationHistory`, and `EmulationRunner` as optional typed fields. Uses `typeof import(...)` references so the underlying modules remain tree-shaken when not provided.
- **`ValidateRuleToolDeps` update** (`validate_rule_tool.ts`): Changes the `guardrails` field from required to optional (or absent in PR 1a, added back in PR 3). Prevents build failure when `DetectionEmulationGuardrails` types are not yet in scope.
- **`runValidateRule` handler branches** (`validate_rule_tool.ts`): Three conditional paths — (1) `enrichments.scoreConfidence` absent → return raw `tp`/`fp` counts; (2) `enrichments.createEmulationHistory` absent → skip persistence, omit `report_id`; (3) `enrichments.EmulationRunner` absent → return `{ error: 'feature_not_available' }` for `real_execution` mode. Each branch produces a valid, typed response.
- **Guardrails construction gate** (`plugin.ts:659`): Wraps `createDetectionEmulationGuardrails(config, logger)` in `if (logInjection || realExecution)`. When both flags are `false` the class property (`this.detectionEmulationGuardrails`, declared at `plugin.ts:219`) is never allocated; downstream callers (`initRoutes`, `registerAgentBuilderAttachmentsAndTools`) receive `undefined` and must handle it.
- **Route registration gate** (`register_routes.ts:39-44`): `validateRuleRoute` registration is wrapped in `if (logInjection)` (PR 1a). `runEmulationCommandRoute` and `haltEmulationRoute` are wrapped in `if (realExecution)` (PR 3). Handler-level flag checks (`getDetectionEmulationFeatureFlags` at request time) remain as defense-in-depth.
- **Per-family tool registration gate** (`detection_emulation_skill.ts:37-157`): `getInlineTools()` filters the tool list based on flag state. When `realExecution` is `false`, only `createValidateRuleTool` is included. The four `run*Command` tools and `createGetEmulationHistoryTool` are added in PRs 3 and 2 respectively, gated by their own flags.
- **Evals package** (`kbn-evals-suite-detection-emulation/`): Ships with PR 1a. The 680-line `validate_rule.spec.ts` runs against the PR 1a response shape (raw TP/FP, no `confidence` or `report_id`). Eval dataset (`validate_rule_dataset.ts`) must be compatible with the simplified schema.

## Data Model

### PR 1a — Tool Response (no enrichments)

```typescript
interface ValidateRuleResultV1 {
  success: boolean;
  scenario_id: string;
  rule_id: string;
  mode: 'log_injection';
  tp: number;
  fp: number;
  matched_signals: string[];
  unmatched_signals: string[];
  poll_duration_ms: number;
}
```

### PR 2 — Tool Response (scoring + history enrichments wired)

```typescript
interface ValidateRuleResultV2 extends ValidateRuleResultV1 {
  report_id: string;
  confidence: number;   // 0–1
  coverage: number;     // 0–1
  precision: number;    // 0–1
  caveats: string[];
}
```

`report_id` is absent in V1. The evals spec must assert presence/absence conditionally, or the PR 2 dataset ships updated expected shapes.

### `ValidateRuleEnrichments` Interface

```typescript
interface ValidateRuleEnrichments {
  scoreConfidence?: typeof import(
    '../../../lib/detection_emulation/confidence_scorer'
  ).scoreConfidence;
  createEmulationHistory?: typeof import(
    '../../../lib/detection_emulation/emulation_history'
  ).createEmulationHistory;
  EmulationRunner?: typeof import(
    '../../../lib/detection_emulation/execution/runner'
  ).EmulationRunner;
}
```

The `typeof import(...)` pattern avoids importing the module at the declaration site. TypeScript resolves the type without executing the module; the runtime caller passes the actual function reference only when the module is available.

### `ValidateRuleToolDeps` — PR 1a Shape

```typescript
interface ValidateRuleToolDeps {
  // guardrails removed in PR 1a; re-introduced as optional in PR 3
  enrichments?: ValidateRuleEnrichments;
  // ... unchanged fields (esClient, soClient, logger, config, etc.)
}
```

The research confirms `guardrails` is currently a required field at `validate_rule_tool.ts:83`. Making it absent (PR 1a) or optional (PR 3) is the minimum change needed to allow independent type-checking of each slice.

## Failure Modes

- **`guardrails` is `undefined` when `realExecution` routes call it**: If `plugin.ts` correctly gates construction behind the feature flag, `guardrails` will never reach `initRoutes` when `realExecution` is `false` because the `run_command` route is also gated. If the two gates get out of sync (flag on, guardrails not constructed), the route handler receives `undefined` where a non-optional object is expected. Mitigation: the route handler must null-check `guardrails` at the start and return 503 with a descriptive error if it is absent. This is defense-in-depth against misconfiguration, not a normal path.
- **Eval spec asserts `confidence` field that PR 1a does not return**: The evals package ships with PR 1a (`validate_rule.spec.ts:680` lines). If the spec's expected shape includes `confidence` or `report_id`, the eval will fail against the V1 response. The spec must be written to match the PR 1a response shape, or the dataset's `expectedOutput` must not assert those fields. This is a correctness risk that must be validated before PR 1a merges.
- **`detection_emulation_skill.ts` registers history tool when `realExecution` is `false`**: The research confirms all six tools are currently registered unconditionally (`detection_emulation_skill.ts:37-157`). The `get-history` tool depends on PR 2's emulation history module. If PR 2 lands but `detectionEmulationLogInjection` is `false`, the tool is registered but its dependencies may behave unexpectedly. Mitigation: gate `createGetEmulationHistoryTool` behind `logInjection` since history is a log-injection-mode artifact, not a real-execution artifact.
- **Type-check isolation fails across slice boundaries**: PRs are stacked (each targets the previous branch). If type errors in PR 1a's enrichment refactor reference types from modules that do not exist until PR 2 or PR 3, the `--project tsconfig.json` scoped type check will fail. The `typeof import(...)` pattern in `ValidateRuleEnrichments` mitigates this only if the imported modules exist as type stubs. If `confidence_scorer.ts`, `emulation_history.ts`, or `runner.ts` do not exist in the PR 1a branch, the `typeof import(...)` references will fail to resolve. Resolution: either (a) introduce empty stub modules in PR 1a with the correct exported type signatures, or (b) declare the optional fields using locally-defined function signatures instead of `typeof import(...)`.
- **Route handler-level flag checks removed as "redundant"**: The existing handler-level `getDetectionEmulationFeatureFlags` guards are defense-in-depth. If a future refactor removes them as redundant after registration-level gating is added, a misconfiguration (flag toggled mid-deploy, config cache stale) could expose the route. Both layers must be preserved.
- **`additional_toolbar_controls.tsx` import of `EmulationFilter` does not exist in PR 1a–3**: The research confirms no emulation UI components exist yet. PR 4 introduces them. No action required before PR 4, but the import must not be added speculatively in earlier PRs — a dead import of a nonexistent module will fail the build.

## Alternatives Considered

- **Single monolithic PR**: Eliminates the enrichment-pipeline refactor entirely and the multi-PR coordination overhead. Rejected because the full feature is ~19k lines across four capability layers; review latency, rollback granularity, and blast radius on flag-off regression all argue for independent merge points.
- **Dynamic `require()` instead of optional enrichment fields**: The enrichments could be loaded via `require()` at call time, falling back to no-op when the module does not resolve. Rejected because dynamic `require()` defeats static analysis, breaks tree-shaking, and introduces runtime module-resolution errors that are harder to surface than type errors at build time.
- **Separate plugin for each slice**: Each PR could live in its own Kibana plugin, communicating via contracts. Rejected by the anti-overengineering gate: the coupling is internal to `security_solution`; a new plugin boundary introduces setup/start lifecycle coordination, contract versioning, and CODEOWNERS surface area without solving a cross-team ownership problem. The enrichment-pipeline pattern achieves the same isolation with in-place code.
- **No-op stub implementations instead of optional fields**: `scoreConfidence` could be replaced by a stub that returns `{ tp: 0, fp: 0 }` rather than making the field optional. Rejected because stubs hide the absence of a capability behind silent behavior; the optional field pattern makes capability presence explicit and testable.
- **Feature flags on the enrichment injection site rather than on tool registration**: Rather than gating which tools are registered in the skill, the flags could be checked inside `getInlineTools()` to filter the returned array. This is equivalent to the chosen approach and equally valid. The proposal uses registration-time filtering; this design follows that framing for consistency with the existing `register_skills.ts` gate pattern (`register_skills.ts:91-104`).
