## Why

The agent builder currently operates in a single implicit mode: the agent receives a user message, reasons about it, uses tools, and responds. There is no mechanism for the agent to construct and follow a structured plan before (or while) executing multi-step tasks. Complex tasks often require upfront planning — breaking work into steps, getting user buy-in, tracking progress, and adjusting as new information emerges. Without planning support, users must manually decompose tasks and track progress themselves, and the agent has no persistent memory of what it set out to accomplish within a conversation.

By introducing a planning mode, we give users a way to engage the agent in a structured, plan-driven workflow: the agent creates a plan, displays it in the UI for the user to review and approve, and then — once the user transitions to agent mode — executes the plan step by step, updating progress in real time. This mirrors patterns found in modern AI coding assistants (e.g. Cursor's plan mode) and is the first step toward richer agent execution modes.

## What Changes

- **Agent modes concept**: Introduce the concept of agent execution modes. The current behavior is named `agent` mode (the execution mode). A new `planning` mode is added for plan creation and refinement. Modes are selected per-conversation-round via a new `agent_mode` parameter on the converse API. The parameter is threaded through the full execution stack, including the Task Manager path (serialized into `AgentExecutionParams` → execution document → deserialized by the task handler).
- **Planning tools**: Three new built-in tools — `create_plan`, `update_plan`, and `suggest_planning_mode` — plus one read-only discovery tool `list_available_tools`. `create_plan` and `update_plan` are available in both modes with mode-dependent behavior. In planning mode, `create_plan` produces a plan that requires user approval. In agent mode, `create_plan` produces a self-planned plan that the agent immediately executes without approval. Planning tools are conditionally included via `selectTools` (the same pattern as filestore tools), not via `ToolAvailabilityContext` which doesn't carry mode information.
- **Read-only skill/tool discovery in planning mode**: In planning mode, the agent has read-only access to skill content and tool descriptions. This allows the agent to inspect available skills and tools before defining the plan, leading to more informed and accurate plans. The agent can read skill files from the skills filesystem and query tool metadata via `list_available_tools`, but cannot execute any tools or trigger side effects.
- **Plan approval flow**: In planning mode, the agent creates and refines the plan but does **not** execute it. The agent can ask the user follow-up questions to clarify requirements. When the plan is finalized (status: `ready`), the user approves it via an explicit "Approve & Execute" button or by manually toggling the mode selector. There is no auto-mode-switch on chat reply — replying in planning mode always stays in planning mode to avoid accidental execution.
- **Plan data structure**: A plan is a first-class concept composed of a title, optional description, an ordered list of action items (each with a description, a granular status, and optional references to the skills/tools the step relies on), a lifecycle status (`draft` or `ready`), and a source (`planning` or `agent`) that determines the approval flow. Action items use a four-state status (`pending` | `in_progress` | `completed` | `failed`) rather than a simple boolean, enabling richer progress tracking in the UI.
- **Skill/tool badges on plan items**: Each action item in a plan can reference which skills and tools it depends on. These references are displayed as badges in the plan panel UI (e.g. `Alert triage skill`, `platform.search tool`), giving the user a clear picture of what capabilities each step will use.
- **Smart mode suggestion**: When the user sends a complex, multi-step request in agent mode, the agent calls the `suggest_planning_mode` tool which emits a `mode_suggestion` streaming event. The UI surfaces this as a dismissable inline banner with a "Switch to Planning Mode" button. The user can accept (switch to planning) or dismiss (stay in agent mode). Using a tool (rather than raw event emission) is more reliable since LLMs can only trigger side effects through tool calls.
- **Agent-mode self-planning (no approval required)**: If the user stays in agent mode for a complex task, the agent can still create a plan using `create_plan`. This "self-planned" plan does not require user approval — the agent immediately begins executing it. The plan is still displayed in the UI panel so the user can track progress, but there is no draft/ready gate or "Execute Plan" button. This allows the agent to stay organized without blocking execution on user approval.
- **Plan-aware agent mode**: When in `agent` mode with an existing plan (whether user-approved from planning mode or self-planned), the agent receives plan-aware instructions that direct it to execute the plan steps in order, updating progress as it goes. Without a plan, agent mode behaves exactly as the current default behavior.
- **Plan as conversation state**: The plan is stored as part of `ConversationInternalState` so it persists across rounds. Dedicated streaming events (`plan_created`, `plan_updated`) are emitted to allow the UI to display and update the plan in real time.
- **Plan UI — collapsible sidebar panel**: The plan is displayed in a collapsible right sidebar panel (not inline in the conversation). This allows the user to see both the conversation flow and the plan progress simultaneously. The panel supports item-level interaction: clicking a plan item pre-fills the chat input with a reference to that step for targeted feedback.
- **Mode selection UI**: A segmented control (`EuiButtonGroup`) allows users to switch between `agent` and `planning` mode when composing a message. Both options are always visible (no dropdown). This control is only visible when the experimental feature flag is enabled.
- **Feature gating**: The entire feature is gated behind the existing `agentBuilder:experimentalFeatures` UI setting. When disabled: no mode selector in the UI, and the `agent_mode` API parameter cannot be set to anything other than `agent`.

## Capabilities

### New Capabilities
- `agent-modes`: Infrastructure for agent execution modes — the `AgentMode` type, mode parameter on the converse API, mode-based routing in the agent execution flow, Task Manager threading, plan approval/mode-transition logic, smart mode suggestion, and feature-flag gating.
- `planning-tools`: The `create_plan`, `update_plan`, `suggest_planning_mode`, and `list_available_tools` tools, planning mode system instructions, plan-aware agent mode instructions, and the plan data structure with lifecycle status.
- `plan-ui`: UI components for the segmented mode selector, the collapsible sidebar plan display panel with approval/execute controls, mode suggestion banner, item-level interaction, and real-time plan update rendering driven by streaming events.

## Impact

- **Data model**: `ConversationInternalState` gains a `plan?: Plan` field (including `status: 'draft' | 'ready'`, `source: 'planning' | 'agent'`, and items with `ActionItemStatus`). New `ChatEventType` values (`plan_created`, `plan_updated`, `mode_suggestion`) are added to the streaming event enum.
- **Packages affected**:
  - `@kbn/agent-builder-common` (`x-pack/platform/packages/shared/agent-builder/agent-builder-common/`) — `AgentMode` type, plan types (`Plan`, `PlanActionItem`, `PlanStatus`, `PlanSource`, `ActionItemStatus`), new chat event types (including `mode_suggestion`), updated `ConversationInternalState`, updated `ChatRequestBodyPayload` with `agent_mode` field.
  - `@kbn/agent-builder-server` (`x-pack/platform/packages/shared/agent-builder/agent-builder-server/`) — updated `ExperimentalFeatures` with `planning` flag.
  - `agent_builder` plugin server (`x-pack/platform/plugins/shared/agent_builder/server/`) — new `planning` mode handler under `modes/planning/`, plan-aware instructions for agent mode under `modes/default/`, planning tool definitions, mode routing in `runAgent`, updated converse route schema, `agent_mode` threaded through `AgentExecutionParams` and Task Manager execution path.
  - `agent_builder` plugin common (`x-pack/platform/plugins/shared/agent_builder/common/`) — updated `ChatRequestBodyPayload` with `agent_mode` field.
  - `agent_builder` plugin public (`x-pack/platform/plugins/shared/agent_builder/public/`) — segmented mode selector component, collapsible sidebar plan display panel with approval controls, mode suggestion banner, item-click interaction, event handling for `plan_created`/`plan_updated`/`mode_suggestion` in `useSubscribeToChatEvents`.
- **Runtime behavior**: `runAgent()` dispatches to either `runDefaultAgentMode` (agent mode) or `runPlanningAgentMode` based on the `agent_mode` parameter. Planning mode uses the same graph building blocks (LangGraph research → answer) but with its own tool selection and prompt factory. In agent mode with a plan, the agent executes the plan and updates progress. The plan state is persisted in `ConversationInternalState` and restored when continuing a conversation.
- **Task Manager path**: `AgentExecutionParams` gains `agentMode?: AgentMode`. The value is serialized into the execution document and deserialized by the Task Manager handler, ensuring mode is preserved across the TM hop.
- **Migration**: No data migration needed. The `plan` field on `ConversationInternalState` is optional. Existing conversations have no plan and operate in `agent` mode. The `agent_mode` parameter defaults to `agent`.
- **APIs**: The converse API (`POST /api/agent_builder/converse` and `/converse/async`) gains an optional `agent_mode` parameter. No new API routes are introduced.
