TSDS older data ingestion simplification

Walkthrough of the proposal to lazily create past backing indices in Time Series Data Streams, and the two follow-on options for simplifying historical backfill.

Overview

Time Series Data Streams (TSDS) currently reject any document whose @timestamp falls outside the time range of an existing backing index. That single rule blocks two everyday workflows:

  • TSDS initialisation — loading seed data into a brand-new stream that has no backing index covering those timestamps.
  • Backfilling — loading historical data into an existing stream whose backing indices only cover recent time ranges.

This document summarises the proposal to unblock initialisation by lazily creating past backing indices, the constraints that apply, and two candidate follow-up options for the harder backfill problem.

Backfill is treated separately because it requires the cluster to scale to the total historical dataset, not just to the ongoing ingest rate. The proposals below keep that scaling trade-off explicit.

Core proposal: lazy past backing index creation

When a document arrives with a timestamp inside the stream's retention writable window, and no backing index yet covers that timestamp, Elasticsearch automatically creates a new backing index for that range and routes the document to it. No manual intervention is needed from the user.

Unblocked

TSDS initialisation

Loading seed data into a freshly created TSDS no longer requires pre-creating a backing index. The first ingested document creates the index covering its timestamp implicitly.

Scope

Only the writable retention window

The behaviour applies inside the retention writable window. Documents older than that window remain rejected, and read-only backing indices are not modified.

Constraints

The proposal explicitly preserves a few invariants so that operators retain a predictable model of the cluster.

Invariants preserved by the lazy backing index behaviour.
ConstraintBehaviour
Read-only backing indicesUnaffected. Documents whose timestamps fall inside a read-only period are still rejected.
Lifecycle ageRelative to the data (the @timestamp), not to the index creation time. A lazily created past index inherits the lifecycle position implied by its data range.
Cluster capacityUsers remain responsible for storage, downsampling, and searchable snapshot planning. Lazy creation does not allocate capacity for them.

What about backfill?

The core proposal deliberately does not address bulk historical backfill. Backfilling a full historical dataset forces the cluster to scale linearly with the total size of that history, not just with the live ingest rate.

Scaling trade-off. If lazy creation absorbed unrestricted historical writes, a cluster sized for steady-state ingest could be overwhelmed by a single backfill job. The runbook approach below keeps the cluster scaled to the backfill rate rather than the backfill volume.

The current manual backfill runbook still applies. The two options below propose ways to make that runbook materially simpler without changing its scaling story.

Option 1 — temporary backfill TSDS

The lighter-touch option. The user still moves backing indices between two streams, but they no longer have to create each backing index by hand.

  1. 1Create a temporary backfill data stream with a fixed start time and no lifecycle.
  2. 2Migrate historical data into the backfill stream in sliding windows.
  3. 3As each window finishes, move its completed backing indices from the backfill stream to the target stream via the Modify Data Stream API.
  4. 4For ILM-managed targets, set index.lifecycle.indexing_complete on the moved index and assign the target ILM policy.
Investment

Smaller engineering lift

This option leans on existing primitives (a second stream plus the Modify Data Stream API). The behavioural change for users is removing the manual create-the-backing-index step from the runbook.

Option 2 — backfill configuration on the target stream

The heavier option. Backfill happens directly into the target stream, gated by a declared backfill window. No index movement is required.

A new data stream setting pauses lifecycle actions on past backing indices created during a declared backfill window.

Setting shape

Declared backfill window

The configuration describes:

  • Start and end time of the backfill window.
  • Optional: expected document count and settings overrides.

Implementation complexity. This option requires a new API with a very specific use case and lifecycle semantics that diverge from the steady-state path. The simpler user experience comes at the cost of more machinery in Elasticsearch.

Comparison

Side-by-side summary of the two backfill simplification options.

The choice is between a smaller engineering lift with a multi-step runbook (Option 1) and a larger engineering lift with a single-step user experience (Option 2).
