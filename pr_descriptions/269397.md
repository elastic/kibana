## Summary

Replaces the legacy alert-analysis skill's direct ES queries with a focused inline tool that calls the shared `findRelatedAlerts` service. The result is **−25% mean input tokens and −24% max tokens** (826k vs 1.15M) with quality metrics at parity — validated over 15-repetition evals (75 total runs).

| | v1 legacy | **v3 (this PR)** |
| ----------------------- | --------: | ---------------: |
| Input mean | 257,998 | **193,219** |
| Input median | 96,191 | **83,404** |
| Input max | 1,152,906 | **826,317** |
| Tool calls mean | 6.2 | **4.7** |
| Tool calls max | 21 | **13** |
| Groundedness | 0.87 | 0.85 ≈ |
| Sequence accuracy | 1.00 | **1.00** |

## What changed

- **`alertAnalysisInlineApiToolSkill`** (new, feature-flagged): inline tool schema `{ alertId, timeWindowHours, hostNames?, userNames?, sourceIps?, destIps? }` calling `findRelatedAlerts` directly; response trimmed to `{ message, sourceEntities, relatedAlerts }` only; `maxResults: 50`
- **Entity shortcut params** (`hostNames`, `userNames`, `sourceIps`, `destIps`): allow the model to skip a `security.alerts` round-trip when entity values are already in context. Removing them caused runaway tail behavior (max tool calls → unbound) in evals — **tool call efficiency, not schema size, is the dominant token driver**
- **`maxResults: 50`** (matches legacy): counter-intuitively better than 25 — more initial context means fewer follow-up tool calls (mean 4.7 vs 5.3 at maxResults=25)
- Removes v2 (#269388) artifacts: `alert_analysis_skill_api_driven.ts`, `evals_alert_analysis_api` Scout config, `alertAnalysisApiDrivenSkill` feature flag
- Adds `evals_alert_analysis_legacy` Scout config for v1 baseline evals
- 11 unit tests passing

## How we got here

Three variants were evaluated before settling on this approach:

- **v2 (#269388)**: skill used `workflow_execute_step` with a `kibana.request` YAML envelope — model had to author YAML on every call, adding ~600–800 tokens of boilerplate overhead and degrading quality (Factuality −0.06, Relevance −0.06). Abandoned.
- **v3 initial** (3-rep estimate, inline tool via HTTP route): 179k mean — promising, but noisy at 3 reps.
- **v4 (#269443)**: dual-register pattern — `findRelatedAlerts` as both inline tool and native Workflows step (`security.getRelatedAlerts`). Mean tokens ballooned to 254k (matching legacy) because the Workflows step returns surplus metadata the model has to reason through. The native step pattern is correct architecture for future Workflows consumers — **if one emerges, v4 is the reference implementation**.

## Reproducing the evals

1. **Start Scout** — from repo root (`yarn kbn bootstrap` if needed):
   - **v3** (this PR):  
     `node scripts/scout.js start-server --arch stateful --domain classic --serverConfigSet evals_alert_analysis_inline_api_tool`  
     Enables **`alertAnalysisInlineApiToolSkill`** + **`agentBuilder:experimentalFeatures=true`** (see [`classic.stateful.config.ts`](src/platform/packages/shared/kbn-scout/src/servers/configs/config_sets/evals_alert_analysis_inline_api_tool/stateful/classic.stateful.config.ts)).
   - **v1 legacy baseline**:  
     `node scripts/scout.js start-server --arch stateful --domain classic --serverConfigSet evals_alert_analysis_legacy`  
     Agent Builder experimental UI on, no inline-tool flag (see [`classic.stateful.config.ts`](src/platform/packages/shared/kbn-scout/src/servers/configs/config_sets/evals_alert_analysis_legacy/stateful/classic.stateful.config.ts)).
2. **Run evals** (Kibana is usually `:5620`, ES `:9220` in these configs):

```bash
TRACING_ES_URL=http://elastic:changeme@localhost:9220 \
KIBANA_URL=http://elastic:changeme@localhost:5620 \
EVALUATION_CONNECTOR_ID=<connector> \
EVALUATION_REPETITIONS=3 \
TEST_RUN_ID=my-alert-analysis-run \
node scripts/evals run --suite agent-builder \
  --grep "Alert Analysis" \
  --model <model>
```

`EVALUATION_CONNECTOR_ID` must exist on the target Kibana (see `node scripts/evals run` / vault config). Point `TRACING_ES_URL` at the same ES cluster Scout uses (`:9220`) so EDOT traces land in the right index.
