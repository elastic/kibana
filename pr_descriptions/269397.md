## Background

The legacy alert-analysis skill (`alertAnalysisSkill`) correlates alerts by running direct Elasticsearch queries inside the tool handler — up to 50 results, with a compact `_source` projection. This works, but the implementation lives entirely in the skill layer with no shared service abstraction.

Two experimental approaches were explored before landing here:

- **v2 (#269388)** — The skill used `workflow_execute_step` with a `kibana.request` YAML envelope to call the correlation API. The model had to author YAML on every invocation, which added ~600–800 tokens of boilerplate overhead per turn and degraded quality metrics (Factuality −0.06, Relevance −0.06, Groundedness −0.01). Abandoned.
- **v4 (#269443)** — Introduced a dual-register pattern: the `findRelatedAlerts` service exposed as both an Agent Builder inline tool AND a native Workflows step type (`security.getRelatedAlerts`). The Workflows step registration is correct architecture for future consumers, but no active Workflows consumer exists yet. Keeping it in scope now adds surface area without benefit. **If a Workflows consumer emerges, the native step pattern from v4 is the reference implementation to follow.**

## What this PR does

Replaces the legacy skill's direct ES query with a focused inline tool that calls the shared `findRelatedAlerts` service.

**Code changes:**
- Adds `alertAnalysisInlineApiToolSkill` — feature-flagged with `alertAnalysisInlineApiToolSkill` experimental flag
- Inline tool schema: `{ alertId, timeWindowHours, hostNames?, userNames?, sourceIps?, destIps? }` — entity shortcut params allow the model to skip a `security.alerts` round-trip when it already has entity values in context
- `findRelatedAlerts` service returns `{ message, sourceEntities, relatedAlerts }` only — no surplus metadata
- `maxResults: 50` matching legacy behavior (see eval findings below)
- 2-way feature flag: `alertAnalysisInlineApiToolSkill` → inline tool; fallback → legacy skill
- Removes v2 artifacts: `alert_analysis_skill_api_driven.ts`, `evals_alert_analysis_api` Scout config, `alertAnalysisApiDrivenSkill` feature flag
- Adds `evals_alert_analysis_legacy` Scout config for running v1 baseline evals
- 11 unit tests updated and passing

## Key findings from 15-repetition evals

### Entity shortcut params matter

Schema description for the 4 optional entity params costs ~100 tokens per turn. The payoff: when the model already has entity values from a prior `security.alerts` call, it can pass them directly and skip an entire `security.alerts` round-trip on the next turn. A `security.alerts` response carries full alert `_source` data — potentially 2,000–5,000 tokens that then accumulate in every subsequent turn's context. At 15 reps the effect is measurable: removing the entity params caused significantly worse tail behavior (max tool calls 21 → runaway), while keeping them held the max at 13. **Schema size is not the dominant cost driver. Tool call efficiency is.**

### maxResults: 50 beats maxResults: 25

An earlier cut set `maxResults: 25` on the assumption that fewer results = lower token cost. The 15-rep data showed the opposite: with 50 results the model has enough context to answer correlation questions on the first pass, leading to fewer total tool calls (mean 4.7 vs 5.3), which more than offsets the per-response size increase.
