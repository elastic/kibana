## Ask

Can a focused **inline tool** that delegates to the shared `findRelatedAlerts` service recover the quality regressions seen with the `workflow_execute_step` / YAML approach, while still keeping the business logic centralised in the service layer?

## Our Findings

The inline-api-tool approach **fully recovers the quality regressions** from the YAML/`workflow_execute_step` path and matches or beats the legacy skill on every judge metric. Sequence accuracy and ToolUsageOnly return to exact legacy parity; Groundedness actually improves above legacy (0.91 vs 0.86).

Token spend is ~18% higher than legacy. The gap is not from YAML reasoning (that is gone) but from the `findRelatedAlerts` service returning richer per-alert objects than the legacy skill's raw ES hits. We experimentally confirmed this: (a) removing surplus metadata fields (`totalMatched`, `returnedCount`, `isTruncated`) from the tool response had no measurable effect, and (b) raising `maxResults` from 25→50 caused input tokens to jump by ~47k, confirming that **result volume, not field count, is the dominant cost driver**.

**We recommend the inline-api-tool pattern as the standard approach for skills that need to call internal services**, and flag per-alert payload compaction as the obvious next lever for closing the remaining token gap.

## What this PR does

- Adds **`alertAnalysisInlineApiToolSkill`** (feature flag, default `false`): a new variant of the `alert-analysis` skill that exposes related-alerts correlation as a **focused inline tool** (`security.alert-analysis.get-related-alerts`) whose handler directly calls `findRelatedAlerts` — no YAML, no `workflow_execute_step`.
- Handler delegates to the `findRelatedAlerts` service (extracted in the parent branch), resolving `spaceId` and `alertsIndex` the same way the HTTP route does. Response shape is compact: `{message, sourceEntities, relatedAlerts}` only (surplus metadata fields stripped).
- **`register_skills.ts`**: three-way precedence — `alertAnalysisInlineApiToolSkill` > `alertAnalysisApiDrivenSkill` > legacy.
- Scout config set **`evals_alert_analysis_inline_api_tool`** for local runs.
- Eval spec example 5 updated to assert `security.alert-analysis.get-related-alerts` (drops `workflow_execute_step` / `expectedWorkflowRequestPath`).

## Why this should improve over YAML / workflow_execute_step

The model previously had to reason about and author workflow YAML before making any API call — adding ~28% input tokens and degrading Factuality and Relevance. With an inline tool the model just passes structured JSON params; the framework handles execution. Skill content no longer needs to explain YAML shape, step types, or inline workflow syntax.

**Result:** The inline-api-tool skill matches or exceeds the legacy skill on every judge metric. The residual token delta is structural (richer service output) and addressable independently. **Recommend the inline-api-tool pattern as the standard for Agent Builder skills that call internal services.**

## Reproducing the evals

1. **Scout** — start with the new config:
   `node scripts/scout.js start-server --arch stateful --domain classic --serverConfigSet evals_alert_analysis_inline_api_tool`
   Enables **`alertAnalysisInlineApiToolSkill`** and **`agentBuilder:experimentalFeatures=true`** (see [`classic.stateful.config.ts`](src/platform/packages/shared/kbn-scout/src/servers/configs/config_sets/evals_alert_analysis_inline_api_tool/stateful/classic.stateful.config.ts)).
2. **Legacy baseline**: `node scripts/evals scout` (default **`evals_tracing`** config; no flag).
3. **Run** (Kibana is usually **`:5620`**, ES **`:9220`**):

```bash
TRACING_ES_URL=http://elastic:changeme@localhost:9220 \
KIBANA_URL=http://elastic:changeme@localhost:5620 \
EVALUATION_CONNECTOR_ID=<connector> \
EVALUATION_REPETITIONS=3 \
TEST_RUN_ID=alert-analysis-inline-api-tool-trimmed \
node scripts/evals run --suite agent-builder \
  --grep "Alert Analysis" \
  --model pmeClaudeV45SonnetUsEast1
```

`EVALUATION_CONNECTOR_ID` must exist on the target Kibana. `TRACING_ES_URL` must point at the ES cluster receiving OTLP traces (same cluster Scout uses, typically `:9220`).
