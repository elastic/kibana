## Summary

Replaces the legacy alert-analysis skill's direct ES queries with a focused inline tool that calls the shared `findRelatedAlerts` service. The result is **ŌłÆ25% mean input tokens and ŌłÆ24% max tokens** (826k vs 1.15M) with quality metrics at parity ŌĆö validated over 15-repetition evals (75 total runs).

## What changed

- **`alertAnalysisSkill`** replaced: the legacy direct ES query implementation is replaced by an inline tool calling `findRelatedAlerts` directly; inline tool schema `{ alertId, timeWindowHours, hostNames?, userNames?, sourceIps?, destIps? }`; response trimmed to `{ message, sourceEntities, relatedAlerts }` only; `maxResults: 50`
- **Entity shortcut params** (`hostNames`, `userNames`, `sourceIps`, `destIps`): allow the model to skip a `security.alerts` round-trip when entity values are already in context. Removing them caused runaway tail behavior (max tool calls ŌåÆ unbound) in evals ŌĆö **tool call efficiency, not schema size, is the dominant token driver**
- **`maxResults: 50`** (matches legacy): counter-intuitively better than 25 ŌĆö more initial context means fewer follow-up tool calls (mean 4.7 vs 5.3 at maxResults=25)
- Removes v2 (#269388) artifacts: `alert_analysis_skill_api_driven.ts`, `evals_alert_analysis_api` Scout config
- Adds `evals_alert_analysis_legacy` Scout config for v1 baseline evals
- 11 unit tests passing

## How we got here

Three variants were evaluated before settling on this approach:

- **v2 (#269388)**: skill used `workflow_execute_step` with a `kibana.request` YAML envelope ŌĆö model had to author YAML on every call, adding ~600ŌĆō800 tokens of boilerplate overhead and degrading quality (Factuality ŌłÆ0.06, Relevance ŌłÆ0.06). Abandoned.
- **v3 initial** (3-rep estimate, inline tool via HTTP route): 179k mean ŌĆö promising, but noisy at 3 reps.
- **v4 (#269443)**: dual-register pattern ŌĆö `findRelatedAlerts` as both inline tool and native Workflows step (`security.getRelatedAlerts`). Mean tokens ballooned to 254k (matching legacy) because the Workflows step returns surplus metadata the model has to reason through. The native step pattern is correct architecture for future Workflows consumers ŌĆö **if one emerges, v4 is the reference implementation**.

## Reproducing the evals

1. **Start Scout** ŌĆö from repo root (`yarn kbn bootstrap` if needed):
   - **v3** (this PR, default):  
     `node scripts/scout.js start-server --arch stateful --domain classic --serverConfigSet evals_alert_analysis_inline_api_tool`  
     Enables **`agentBuilder:experimentalFeatures=true`** (see [`classic.stateful.config.ts`](src/platform/packages/shared/kbn-scout/src/servers/configs/config_sets/evals_alert_analysis_inline_api_tool/stateful/classic.stateful.config.ts)).
   - **v1 legacy baseline**:  
     `node scripts/scout.js start-server --arch stateful --domain classic --serverConfigSet evals_alert_analysis_legacy`  
     Agent Builder experimental UI on (see [`classic.stateful.config.ts`](src/platform/packages/shared/kbn-scout/src/servers/configs/config_sets/evals_alert_analysis_legacy/stateful/classic.stateful.config.ts)). Requires checking out the pre-v3 branch to run against the old implementation.
2. **Run evals** (Kibana is usually `:5620`, ES `:9220` in these configs):

```bash
TRACING_ES_URL=http://elastic:changeme@localhost:9220 \
KIBANA_URL=http://elastic:changeme@localhost:5620 \
EVALUATION_CONNECTOR_ID=<connector> \
EVALUATION_REPETITIONS=3 \
TEST_RUN_ID=my-alert-analysis-run \
node scripts/evals run --suite agent-builder \
  --grep "Alert Analysis" \
  --model <model>
```

`EVALUATION_CONNECTOR_ID` must exist on the target Kibana (see `node scripts/evals run` / vault config). Point `TRACING_ES_URL` at the same ES cluster Scout uses (`:9220`) so EDOT traces land in the right index.
