# GitHub Issue: [One Workflow] Step registry improvements for Cases

**Repo:** https://github.com/elastic/security-team  
**Title:** `[One Workflow] Step registry improvements for Cases`  
**Suggested labels:** `enhancement`

---

## Description

Improvements to the workflows step extensions registry are needed to meet Cases team requirements. These changes affect the **selection** handler, the **SelectionContext** passed to selection callbacks, and the **connectorIdSelection** feature (including its documentation).

## Context

- Step extensions plugin README: `src/platform/plugins/shared/workflows_extensions/README.md`
- Types: `src/platform/packages/shared/kbn-workflows/types/v1.ts` (e.g. `ConnectorIdSelectionHandler`, `SelectionContext`, `PropertySelectionHandler`)

## Requirements

### 1. Selection handler for array nested props

**Current behavior:** The selection handler assumes the schema is composed of **objects only**. Property paths use dot notation for nested objects (e.g. `'agent.id'`).

**Needed:** Support **arrays** in the schema—arrays of primitives (e.g. strings) or arrays of nested objects. Each item inside the array should be selectable individually (e.g. per-item entity selection in `string[]` or `SomeObject[]`).

**Impact:** Cases (and other consumers) need to attach selection (search/resolve/getDetails) to array elements, not only to top-level or object-nested properties.

---

### 2. Selection context: include current step `input` and `config` values

**Current behavior:** `SelectionContext` passed to `search`, `resolve`, and `getDetails` only includes:
- `stepType`
- `scope` (`'config' | 'input'`)
- `propertyKey`

**Needed:** Extend the context so callers can use the **current step definition values** for filtering, message generation, etc. Proposed shape:

- **`context.values`**: `{ config: {...}, input: {...} }` — current config and input of the step (typed according to the step's config/input schemas).

This allows selection handlers to depend on other fields of the same step (e.g. filter connector list by category when `config.category` is set, or tailor messages from `input`/`config`).

---

### 3. Dynamic `connectorTypes` in `connectorIdSelection` + documentation

**Current behavior:**
- `ConnectorIdSelectionHandler` in `v1.ts` (around 550–561) only supports a **static** `connectorTypes: string[]`.
- The `connectorIdSelection` feature is **not documented** in the workflows_extensions README.

**Needed:**
- Allow **dynamic** connector types: support an **async function** that returns `Promise<string[]>` (in addition to the static `string[]`), so the list of connector types can depend on runtime/context (e.g. step config, feature flags, permissions).
- **Document** `connectorIdSelection` in the README: what it is, where it applies (e.g. `connector-id` in config schema), and how to use static vs dynamic `connectorTypes`.

Reference in code: `src/platform/packages/shared/kbn-workflows/types/v1.ts` (e.g. lines 489–490 for `connectorIdSelection`, and the `ConnectorIdSelectionHandler` interface with `connectorTypes: string[]`).

## Proposed direction (summary)

1. **Selection + arrays:** Extend the step registry/editor so that selection can be defined for array elements (path semantics and UI for per-item selection).
2. **Selection context:** Add `context.values: { config, input }` to the object passed to `search`/`resolve`/`getDetails`.
3. **connectorIdSelection:** Add support for `connectorTypes` as `string[] | (() => Promise<string[]>)` (or equivalent) and document the feature in the workflows_extensions README.

## Impact

- **Cases team:** Can implement step extensions that need array-level selection and context-aware connector selection and messaging.
- **Other consumers** of the step registry benefit from the same capabilities and from documented `connectorIdSelection`.
