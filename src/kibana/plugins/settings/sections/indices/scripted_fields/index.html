{{ action }} Scripted Field

By default, Elasticsearch scripts use Groovy , which is a lot like Java, but scriptier. We'll let you do some reading on Groovy , but the important bit is that there are three ways to access the value of Elasticsearch fields, each with distinct advantages and disadvantages:

doc['some_field'].value (recommended)

Loads the field value from the field data cache (thus, the value is loaded from the inverted index). Very fast at the cost of memory consumption. Requires the field to be indexed.

_source.some_field

Loads the field value from the _source, which means _source must be enabled. This has the benefit of treating the field value as not_analyzed without needing to reindex, however it is extremely slow, and very disk I/O heavy. It will probably timeout if your timeouts are low and you have lots of data.

_fields['some_field']

Loads the field value from the stored field, if the field is stored. Stored fields are separate from _source fields, and must be defined explicitly in your mapping. If you don't know if a field is stored or not, it probably is not. Might be lighter than _source, yet still very heavy and slow compared with doc notation. You probably don't want this.

Proceed with caution

Scripted fields can be used to display and aggregate calculated values. As such, they can be very slow, and if done incorrectly, can cause Kibana to be unusable. There's no safety net here. If you make a typo, unexpected exceptions will be thrown all over the place!

You already have a field with the name {{ scriptedField.name }}. Naming your scripted field with the same name means you won't be able to query both fields at the same time.
