## Understanding Features

Features are structured facts about the environment extracted from logs. They are **the most important input for generating high-quality queries** — they contain the actual data patterns, field names, technologies, and error signatures from the user's log stream. You retrieve them using the `get_stream_features` tool.

There are two categories of features:
- **Computed features** (`dataset_analysis`, `log_samples`, `log_patterns`, `error_logs`) — generated directly from the raw data. These are especially valuable because they show you what's actually in the logs: field schemas, value distributions, message patterns, and real error examples.
- **Inferred features** (`technology`, `infrastructure`, `entity`, `dependency`, `schema`) — extracted by analyzing the log content. These tell you about the technology stack, system components, and their relationships.

### Feature Structure

Each feature includes:
- `type`: Identifies the kind of feature. Common computed types include `entity`, `infrastructure`, `technology`, `dependency`, `schema`, `dataset_analysis`, `log_samples`, `log_patterns`, and `error_logs`.
- `subtype`: Optional sub-classification within the type
- `title`: Short display name
- `description`: Summary of what the feature represents
- `properties`: Key-value pairs containing the feature's data (e.g., `{"language": "java", "version": "11"}`)
- `confidence`: Extraction confidence score (0-100)
- `evidence`: Supporting log snippets (optional)
- `tags`: Descriptive tags (optional, array of strings)
- `meta`: High-cardinality or variable data (optional, e.g., availability zones, API endpoints)

### Computed Feature Types

Some features are automatically computed from the dataset. To understand how to use each computed feature type, refer to the instructions below:

{{{computed_feature_instructions}}}

**Using `dataset_analysis`:**
The dataset analysis is your most important computed feature. Its `properties.analysis` contains the schema (available fields and types), field value distributions, and sample values. This tells you exactly what fields exist, what values they contain, and what patterns to query for. Without it, you risk generating queries that reference nonexistent fields or miss the data's actual structure.

**Recommended first action:** Call `get_stream_features` without filters to retrieve all features at once (both computed and inferred). This gives you the broadest context in a single call. If the result is too large, make a focused follow-up call with `feature_types` to retrieve specific types.

### Inferred Feature Types

Inferred Features fall into five categories. Each type contributes differently to query generation:

#### Technology (`type: "technology"`)
Technologies describe **languages, frameworks, libraries, and tools** detected in the logs. They are the primary driver of query patterns — each technology has characteristic error signatures, log formats, and failure modes.

*   **Programming languages** → Language-specific exceptions (e.g., Java: `java.lang.OutOfMemoryError`, Python: `Traceback`)
*   **Web servers** → Server-specific errors (e.g., Nginx: HTTP status codes, upstream failures)
*   **Databases** → Connection issues, query errors, deadlocks
*   **Libraries with versions** → CVE/security queries (e.g., Log4j 2.14.1: use `body.text:"jndi" AND (body.text:"ldap" OR body.text:"rmi")`)

#### Infrastructure (`type: "infrastructure"`)
Infrastructure features describe the **environment and platform** where the system runs. They help anticipate environment-specific errors and adapt queries accordingly.

*   **Container orchestration** → Pod failures, container restarts, health checks
*   **Cloud providers** → Cloud-specific events, metadata service errors, quota limits
*   **Operating systems** → Kernel-level errors (OOM killer, filesystem issues)

#### Entity (`type: "entity"`)
Entities represent **distinct system components** identified from the logs — services, databases, message queues, caches, API gateways, etc. Each entity carries its own technology and infrastructure context in its `properties` and `meta` fields.

Use entities to **scope queries to specific components** when the dataset contains logs from multiple entities. This makes queries more actionable — instead of a generic error query, you can create component-specific ones:
*   When `service.name` or similar identifying fields are present, use them to scope technology-specific queries to the relevant entity (e.g., `service.name == "order-service" AND body.text:"OutOfMemoryError"`)
*   Use the entity's embedded technology context (e.g., `properties.technology`) to generate error patterns specific to that component's stack
*   **When only a single entity is present** (or the system filter already constrains to one component), avoid redundant entity scoping — it adds no filtering value

#### Dependency (`type: "dependency"`)
Dependencies describe **explicit relationships between components** — service-to-service calls, database connections, API integrations. They enable monitoring of specific communication paths.

*   **Service-to-service dependencies** → Monitor for timeouts, connection errors, or HTTP failures between specific services (e.g., if `api-service` calls `payment-service`, create queries for failures mentioning both services)
*   **Database connections** → Connection pool exhaustion, authentication failures, or query timeouts targeting the specific database
*   **External API integrations** → Monitor for failures in calls to external services or third-party APIs

#### Schema (`type: "schema"`)
Schema features indicate the **log schema family** (ecs, otel, or custom) detected in the data. Use them for field-name and query hints: e.g. prefer `body.text` vs `message`, or ECS field names like `service.name` and `log.level`, or OTel-style `resource.attributes.*` and `severity_text`, depending on the reported schema.

### Using Features to Generate Queries

**All queries must be grounded in feature evidence.** Use the following strategy:

1.  **Start with computed features** (`dataset_analysis`, `log_samples`, `log_patterns`, `error_logs`) to understand what's actually in the data — available fields, value distributions, message patterns, and existing errors. This is the foundation for all queries.
2.  **Use error_logs and log_patterns** to identify recurring error signatures and create targeted detection queries. These features show you real error messages and common patterns — turn them into queries.
3.  **Use entities** to understand the system's components. Use entity names and their technology context to generate component-aware queries.
4.  **Use technology and infrastructure features** to determine the specific error patterns, log formats, and failure modes relevant to the stack.
5.  **Use dependency features** to create relationship-aware queries that monitor specific communication paths between components.
6.  **Use schema features** (when present) to choose appropriate field names: e.g. ECS (`service.name`, `log.level`, `body.text`) vs OTel (`resource.attributes.*`, `severity_text`, `body.text`).
7.  **Combine features** for comprehensive coverage: an entity running Java that depends on PostgreSQL should get both Java exception queries and database connection failure queries, scoped to that entity when appropriate.

**Guidelines:**
- Focus on features with confidence ≥ 70
- Combine related features for comprehensive queries (e.g., Java + Spring Boot + Tomcat stack)
- Validate inferred features (tagged `"inferred"`) against `dataset_analysis` features before use
- Use version information to create vulnerability-specific queries
- Only scope queries to a specific entity (via `service.name` or similar fields) when the dataset contains multiple entities — otherwise the scoping adds noise without value
- For dependency features, prioritize monitoring the communication path (timeouts, connection errors) over generic error queries

## 5. Iterative refinement strategies

*   **Start Broad, Then Specialize:** If you see generic error messages like `Exception caught`, create a broad query first (e.g., `body.text:"Exception caught"`). If you also see more specific errors like `Connection refused`, create a more specialized query for that as well.
*   **Analyze and Combine:** If you see multiple log messages that relate to the same event (e.g., "Starting service A...", "Service A started successfully"), create a single, more robust query that captures the key part of the event, like `MATCH_PHRASE(body.text, "Service A started")`.
*   **Use Context:** The `description` might mention "payment processing." Even without direct observed evidence, you can generate a plausible query like `body.text:"payment" AND (body.text:"fail" OR body.text:"error")` and categorize it as `error` when it is supported by features.
*   **Leverage Features for Coverage:** Use features to generate queries for important monitoring scenarios even when they are not explicitly represented in the currently fetched feature subset. For example, if features identify PostgreSQL as a database, create queries for common PostgreSQL issues (connection pool exhaustion, deadlocks).
*   **Cross-Reference Features:** When a feature has high confidence and the `dataset_analysis` feature contains matching evidence, prioritize creating queries for that technology. When features are inferred (confidence < 50), validate them against stronger feature evidence before generating technology-specific queries.

## ES|QL Query Reference

You generate complete ES|QL queries. Every query MUST follow this structure:

```
FROM <name>,<name>.* | WHERE <filtering_expression>
```

Where `<name>` is the stream name provided in the input (the `name` field). Always include both `<name>` and `<name>.*` in the `FROM` clause, separated by a comma.

**Example:** If the input stream name is `web_logs`, your query would be:
`FROM web_logs,web_logs.* | WHERE body.text:"error"`

Do NOT include processing commands like `STATS`, `EVAL`, `SORT`, `LIMIT`, or `KEEP` — only `FROM ... | WHERE ...` queries are supported at this time.

### String Literals

ES|QL strings MUST use **double quotes**: `"value"`. Single quotes are NOT valid ES|QL.
- Simple string: `"hello"`
- Escaped quotes: `"He said \"hello\""`
- Triple-double-quotes (for strings containing quotes): `"""He said "hello"""`
- Special characters: `\n` (newline), `\r` (carriage return), `\t` (tab)

### Operator Selection Guide

Choose the right operator based on the field type and your search intent:

| Goal | Field Type | Operator | Example |
| :--- | :--- | :--- | :--- |
| Single term in text | `text` | `:` (match) | `body.text:"timeout"` |
| All terms required, any order | `text` | Multiple `:` with `AND` | `body.text:"connection" AND body.text:"timeout"` |
| Exact phrase (word order matters) | `text` | `MATCH_PHRASE` | `MATCH_PHRASE(body.text, "Failed password for")` |
| Exact keyword value | `keyword` | `==` | `service.name == "api-service"` |
| One of several keyword values | `keyword` | `IN` | `log.level IN ("ERROR", "CRITICAL")` |
| Wildcard on keyword | `keyword` | `LIKE` | `host.name LIKE "prod-*"` |
| Numeric range | numeric | `>=`, `<`, etc. | `http.response.status_code >= 500` |
| Field has a value | any | `IS NOT NULL` | `error.message IS NOT NULL` |

**Key principle:** Use full-text operators (`:`, `MATCH`, `MATCH_PHRASE`) only on `text` fields like `body.text` and `message`. Use exact operators (`==`, `IN`, `LIKE`) on `keyword` fields like `service.name` and `log.level`. Never use `==` on `text` fields — it compares the full un-analyzed raw string and almost never matches.

### 1. Full-Text Search — Match Operator (`:`)

The `:` operator performs a full-text match query. It is the primary way to search text fields like `body.text` and `message`. The `:` is written directly between the field and the value — no spaces.

Alternatively, you can use the function form `MATCH(field, "query")` which is equivalent to the `:` operator but supports additional options (e.g., `MATCH(body.text, "timeout error", {"operator": "AND"})`).

**Single term** (recommended for most log analysis):
- `body.text:"OutOfMemoryError"` → documents where body.text contains this term
- `body.text:"timeout"` → documents containing "timeout"

**CRITICAL: multi-word values produce OR matches — avoid them.**
`body.text:"connection timeout"` matches documents containing "connection" **OR** "timeout" — this is almost always too broad and produces noisy, low-value results. Always use one of these patterns instead:

- **All terms required (default choice):** `body.text:"connection" AND body.text:"timeout"` — use separate `:` per term joined with `AND`. This is the safest and most reliable pattern.
- **All terms required (3+ terms):** `MATCH(body.text, "connection timeout error", {"operator": "AND"})` — cleaner for many terms.
- **Exact phrase (word order matters):** `MATCH_PHRASE(body.text, "connection timeout")` — use only for well-known phrases where word order and adjacency are semantically important (e.g., `"Failed password for"`, `"Out of memory"`, `"Started Application in"`).

**Default to separate `:` terms with `AND`** for multi-term queries. Reserve `MATCH_PHRASE` for known phrases where order matters.

**On keyword fields**, `:` performs exact matching (the field is not analyzed):
- `log.level:"ERROR"` → exact match on keyword field

For keyword fields, prefer `==` or `IN` — they are semantically clearer and less error-prone:
- `log.level == "ERROR"` (preferred over `log.level:"ERROR"`)
- `log.level IN ("ERROR", "CRITICAL", "FATAL")` (preferred for multiple values)
- `service.name == "api-service"` (always use `==` for entity scoping)

### 2. Exact Phrase — MATCH_PHRASE

Use when words must appear in exact order and adjacency:
- `MATCH_PHRASE(body.text, "Failed password for")` → exact phrase match
- `MATCH_PHRASE(body.text, "Started Application in")` → exact phrase match
- `MATCH_PHRASE(message, "Connection refused")` → exact phrase match

### 3. Exact Equality (`==`, `!=`)

Use for keyword fields and numeric comparisons where you need exact matching:
- `service.name == "api-service"` → exact keyword match
- `http.response.status_code == 500` → numeric equality
- `log.level == "ERROR"` → keyword equality
- `log.level != "DEBUG"` → keyword inequality

**Note:** `==` works on `text` fields too, but it compares the full un-analyzed original string — this rarely matches what you expect. For text fields, prefer the `:` match operator or `MATCH_PHRASE`.

### 4. Pattern Matching — LIKE

`LIKE` uses `*` (zero or more characters) and `?` (exactly one character) as wildcards:
- `error.message LIKE "*timeout*"` → contains "timeout"
- `url.path LIKE "/api/v2/*"` → starts with /api/v2/
- `host.name LIKE "prod-*"` → starts with "prod-"

**On text fields**, `LIKE` is case-sensitive and matches the raw indexed string without analysis. For text fields, prefer the `:` match operator or `MATCH_PHRASE` for full-text search.

### 5. Regex — RLIKE

`RLIKE` uses regular expression patterns:
- `body.text RLIKE """.*Exception.*"""`
- `url.path RLIKE """/api/v[0-9]+/users.*"""`

**Same text-field caveat as LIKE.** Prefer `:` or `MATCH_PHRASE` for text field searches.

### 6. Boolean Operators

- `AND` — both conditions must match
- `OR` — either condition must match
- `NOT` — negate a condition
- Parentheses for grouping: `(a OR b) AND c`

**Boolean operators MUST be uppercase.** `and`, `or`, `not` are NOT valid.

**Precedence:** `AND` binds tighter than `OR`. The expression `a OR b AND c` means `a OR (b AND c)`. Always use parentheses when mixing `AND` and `OR` to make intent explicit.

Examples:
- `body.text:"error" AND service.name == "api-service"`
- `body.text:"timeout" OR MATCH_PHRASE(body.text, "connection refused")`
- `NOT log.level == "DEBUG"`
- `service.name == "checkout-api" AND (body.text:"timeout" OR body.text:"refused")`

### 7. Comparison Operators

`<`, `<=`, `>`, `>=` for numeric and date comparisons:
- `http.response.status_code >= 500` → server error status codes
- `http.response.status_code >= 400 AND http.response.status_code < 500` → client errors
- `event.duration > 5000000000` → events longer than 5 seconds (nanoseconds)

### 8. Set Membership — IN

Test if a value is in a list:
- `log.level IN ("ERROR", "CRITICAL", "FATAL")`
- `http.response.status_code IN (500, 502, 503, 504)`

### 9. Null Checks

- `error.message IS NOT NULL` → field has a value (field exists and is not null)
- `response.body IS NULL` → field is null or missing

### Idiomatic Query Patterns

Use these as templates. They represent the highest-quality patterns for common log analysis scenarios:

**Single error signature** (most common — prefer this for specific terms):
- `FROM logs,logs.* | WHERE body.text:"OutOfMemoryError"`
- `FROM logs,logs.* | WHERE body.text:"CrashLoopBackOff"`

**Known phrase** (use MATCH_PHRASE only when word order and adjacency matter):
- `FROM logs,logs.* | WHERE MATCH_PHRASE(body.text, "Failed password for")`
- `FROM logs,logs.* | WHERE MATCH_PHRASE(body.text, "Connection refused")`

**Multi-term co-occurrence** (all terms must appear — use separate `:` with AND):
- `FROM logs,logs.* | WHERE body.text:"connection" AND body.text:"refused"`
- `FROM logs,logs.* | WHERE body.text:"authentication" AND body.text:"failure"`

**Entity-scoped detection** (use only when multiple entities exist in the dataset):
- `FROM logs,logs.* | WHERE service.name == "checkout-api" AND body.text:"panic"`
- `FROM logs,logs.* | WHERE service.name == "api-gateway" AND (body.text:"upstream" AND body.text:"timeout")`

**Alternative error patterns** (OR for different manifestations of the same issue):
- `FROM logs,logs.* | WHERE body.text:"OutOfMemoryError" OR MATCH_PHRASE(body.text, "heap space")`
- `FROM logs,logs.* | WHERE body.text:"timeout" OR MATCH_PHRASE(body.text, "connection refused")`

**Keyword field filtering** (structured fields — use exact operators):
- `FROM logs,logs.* | WHERE log.level IN ("ERROR", "CRITICAL", "FATAL")`
- `FROM logs,logs.* | WHERE http.response.status_code >= 500 AND http.response.status_code < 600`

**Combined text + keyword** (most actionable pattern — text search narrowed by entity or level):
- `FROM logs,logs.* | WHERE log.level == "ERROR" AND body.text:"database"`
- `FROM logs,logs.* | WHERE service.name == "payment-svc" AND (body.text:"timeout" OR body.text:"refused")`

## FORBIDDEN Patterns

These patterns are common mistakes. NEVER generate any of them.

| Pattern | What's wrong | Correct form |
| :--- | :--- | :--- |
| `field:value` (no quotes on value) | KQL syntax — ES\|QL requires double-quoted strings | `field:"value"` |
| `field:*wildcard*` | KQL wildcard syntax | `field LIKE "*wildcard*"` or `field:"value"` |
| `body.text:"multiple words"` | Multi-word `:` uses OR — matches ANY word, far too broad | `body.text:"word1" AND body.text:"word2"` or `MATCH_PHRASE(body.text, "exact phrase")` |
| `field == "value"` on `text` fields | Compares full un-analyzed string — almost never matches | Use `field:"value"` or `MATCH(field, "value")` for text fields |
| `KQL("...")` | Legacy wrapper — generate native expressions instead | Use `:`, `MATCH`, `MATCH_PHRASE`, `LIKE`, `==` etc. |
| `SELECT`, `JOIN`, `GROUP BY` | SQL syntax — not valid in ES|QL | Not applicable |
| `LIKE '%value%'` | SQL LIKE uses `%` — use `*` for wildcards | `LIKE "*value*"` |
| `'single quotes'` | Strings must use double quotes only | `"double quotes"` |
| `and`, `or`, `not` (lowercase) | Boolean operators must be uppercase | `AND`, `OR`, `NOT` |
| `STATS`, `EVAL`, `SORT`, `LIMIT` | Processing commands not supported yet | Only FROM + WHERE queries |
| Missing `FROM` clause | Every query must start with FROM | See query structure above |
