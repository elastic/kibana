## Managed OTLP Endpoint Setup

The fastest way to get application data into Elastic on serverless is the **managed OTLP endpoint** — a fully-managed, hosted OpenTelemetry collector. Elastic auto-provisions the endpoint URL and an API key; there is nothing to deploy or operate.

---

## Step 1 — Open the OpenTelemetry onboarding flow

Send the user to the OpenTelemetry APM onboarding page. On serverless, navigate directly to:

**[Set up OpenTelemetry → APM onboarding](/app/observabilityOnboarding#/otel-apm)**

The page automatically creates an API key (scoped to `apm:event:write`) and displays the managed OTLP endpoint URL for the project.

---

## Step 2 — Configure the OpenTelemetry SDK

Copy the three environment variables shown on the onboarding page and set them in the application's environment (Dockerfile, Kubernetes manifest, CI pipeline, etc.):

| Variable | Value |
|---|---|
| `OTEL_EXPORTER_OTLP_ENDPOINT` | The managed OTLP URL from the page (e.g. `https://<project>.apm.elastic.cloud:443`) |
| `OTEL_EXPORTER_OTLP_HEADERS` | `Authorization=ApiKey <base64-key-from-page>` |
| `OTEL_RESOURCE_ATTRIBUTES` | `service.name=<app-name>,service.version=1.0.0,deployment.environment.name=production` |

Use the exact values shown on the onboarding page — they are unique to the user's project and session.

---

## Step 3 — Install the Elastic Distribution of OpenTelemetry (EDOT) SDK

Use the EDOT SDK for the application's language — it comes pre-configured for Elastic and requires less boilerplate than the upstream OTel SDK.

Available for: **.NET · Java · Node.js · PHP · Python · Android · iOS**

Full SDK guides and quickstarts: https://ela.st/edot-sdks

If the user already has an upstream OTel SDK, they only need to update the exporter to point at the managed OTLP endpoint — no migration required.

---

## Step 4 — Verify data is flowing

After restarting the application, the onboarding page polls for incoming data. Once traces, logs, or metrics arrive, it shows a confirmation and links directly to the APM Service Inventory.

If no data appears after a few minutes:
1. Confirm the three env vars are set in the **running** process, not just the shell.
2. Check application logs for OTLP exporter errors (connection refused, auth failures).
3. Verify the application is receiving traffic (send a test request).
4. See the managed OTLP endpoint docs for troubleshooting: https://www.elastic.co/docs/reference/opentelemetry/motlp

---

## Notes

- On **serverless**, the managed OTLP endpoint is GA — no feature flags needed.
- The API key created by the onboarding flow has minimal scope (`apm:event:write`) and is safe to embed in application config.
- If the user asks about a specific language, offer to walk through that language's SDK configuration in detail.
