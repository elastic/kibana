## Significant event remediation

You are helping after **Observability significant event** work has already produced a structured finding. The canonical document is loaded with **`get_sigevent_event`** (attachment-bounded). Add **live telemetry that answers new questions** the attachment does not fully answer (fresh time, different dimension, caller/dependency health, a second rule). Do **not** cold-start a full RCA, fire unrelated `ki_search`, or re-run the **same** evidence pipeline many times.

---

## Default investigation ladder (follow in order)

1. **`get_sigevent_event`** — Call **once** if you do not already have the full `_source` payload in this turn.
2. **One validation `execute_esql`** when `evidences[].esql_query` exists:
   - Pick **one** query: prefer an evidence row with `confirmed: true`, else the latest by `collected_at`.
   - **Dedupe** identical `esql_query` strings — run each distinct pipeline **once** only.
   - Use a **short** `time_range` (e.g. `now-2h` to `now` unless the user specified otherwise). Respect existing `LIMIT`; keep row counts small.
3. **Complementary expansion** (next section) — when its conditions apply, you **must** run it before the final answer. This is **not** optional thoroughness; it is how you obtain **orthogonal** live data.
4. **Synthesize**: prior findings from the document, delta from step 2, **new** facts from step 3 when you ran it, then prioritized remediation from `recommendations[]` adjusted only when tools justify it.
5. **Remediation workflow (mandatory)** — After steps 1–4, you **must** call **`observability.remediation_workflow`** **exactly once** in a **dedicated tool round**, **alone**. Do **not** batch it with step 2, step 3, complementary tools, `ki_search`, or step 6. Pass a single **`query`** string that clearly states **`service_name`**, **`namespace`**, remediation **`reason`** / context, **and an ordered remediation plan derived from `recommendations[]`**. The server always runs **Incident Remediation — Standard Coordination** (records scope, coordination window, then closure). Phrase `query` so those fields — including the plan — are obvious from the text. If any `recommendations[]` entry mentions **rollback / revert / redeploy a previous revision**, place that as the **first** plan step. Otherwise, paraphrase each recommendation into one short imperative step, preserving the order from the document.

   Kibana will **prompt the user for confirmation** before the workflow runs (in-handler HITL, same pattern as `observability.elasticsearch`): the tool resolves the workflow, **interrupts** for confirmation, then **resumes** on the next agent step after the user accepts. Continue the conversation via **`/api/agent_builder/converse`** and supply the pending prompt id under `prompts` when the round is awaiting confirmation; in standalone / non-interactive mode the call is declined. Derive scope from the strongest validated evidence (`service.name`, `dependency_edges`, `recommendations[]`, APM/tool output, or user text). If `namespace` is absent from evidence, use **`default`** unless the event or user clearly names another; if the focal service is ambiguous, pick the single primary failing service from the attachment and reflect that choice in `query`.
6. **Post-remediation verification (mandatory, always last)** — After step 5 has resolved (HITL accepted, declined, or the workflow returned a status), you **must** produce **two artifacts** in **one or two dedicated tool rounds** (never batched with step 5, step 2, complementary tools, or `ki_search`) so the user can decide whether anything has changed since the remediation action. Use the **same** evidence row that step 2 validated (prefer `confirmed: true`, else latest by `collected_at`); do **not** switch to a different evidence row here.

   - **Area chart** — Call **`platform.core.create_visualization`** **exactly once** with `chartType: "XY"`. The natural-language `query` must request an **XY area chart** showing the significant-event signal **over time**, default window **`now-2h` to `now`**, with an appropriate time bucket and **no omitted empty intervals**. Pass the promoted evidence ES|QL **as intent only** — preserve data sources, filters, and `MATCH` semantics, but **adapt** it into a time-bucketed aggregation; do **not** paste it verbatim when it carries `LIMIT`, a sample-only window, or a non-trend shape. If the first attempt does not return a visualization, you may retry **once** with a tightened minimal XY time series (bucket by `@timestamp`, one `COUNT` measure, strict window).
   - **Last-seen evidence query** — Run **`platform.core.execute_esql`** **exactly once** to surface a **single** last-seen `@timestamp` (or “no matches in window”). Either approach is acceptable: (a) **adapt** the evidence ES|QL into `STATS MAX(@timestamp)` while preserving filters / `MATCH` semantics and dropping `LIMIT`, **or** (b) **rerun** the original evidence ES|QL string against a narrow post-remediation window (e.g. `now-30m` to `now`) and take the latest row’s `@timestamp`. Pick one path; do not run both.

## Complementary expansion (orthogonal live data)

Run this **after** step 2, **in separate tool rounds** (never batch step 2 with these in one round). Each call must target information the **attachment did not already contain** (e.g. narrative + sample ES|QL in the doc; you add **service health**, **caller/checkout**, **second rule**, or **one** distinct ES|QL — not the same query again with no new intent).

**When validation shows the issue is still active** (or the user asked to verify, extend, or assess blast radius), perform **complementary expansion** if **any** trigger below applies. Use **up to three** complementary tool calls total from the bullets (pick what applies; skip bullets that do not apply):

- **`dependency_edges` is non-empty** — You **must** call **`get_services`** scoped to the **`source` and `target`** service names from those edges (one combined filter or at most two calls). Optionally **`get_service_topology`** for one of those services **only if** it adds a fact **`get_services` did not**. If topology returns no meaningful connections, do not chain more topology or **`get_traces`** for the same question unless the user expands scope.
- **`recommendations[]` implies another scope** (e.g. “query checkout …”, “blast radius on checkout”) — You **must** call **`get_logs`** or a **narrow `execute_esql`** for that scope. The query **must differ** from the evidence ES|QL in step 2 (different filters, entity, or aggregate — e.g. checkout errors or error-rate buckets, not another copy of the same LIMIT-5 evidence pipeline).
- **More than one `rule_names` entry** and the evidence ES|QL only reflects **one** rule — Add **one** additional **`execute_esql`** aligned to the **other** rule’s intent (bounded `LIMIT`), not a rerun of the same string as step 2.
- **User explicitly asks for knowledge indicators or alerts** — After the bullets above that apply, you may call **`ki_search`** once with **`stream_names`**, **low `limit`**, and **`search_text`** built only from **`rule_names`** + **`cause_kis[].name`** (no broad semantic blur).

If the user only wanted a quick read of the attachment **and** validation shows **resolved** (no ongoing failure class), you may **skip** this entire section and answer from the document + step 2 only.

## When to use this skill

- A significant-event attachment is present, or the user refers to that event after you loaded it.
- The user wants remediation, validation, or next steps.

The document already carries `root_cause`, `summary`, `evidences`, `stream_names`, `cause_kis`, `dependency_edges`, and `recommendations` when present — treat them as **prior evidence**, not something to re-discover from scratch.

---

### No parallel validation + expansion

Do **not** batch the **first** validation **`execute_esql`** with **`ki_search`**, **`get_services`**, **`get_service_topology`**, **`get_traces`**, **`get_logs`**, **`observability.remediation_workflow`**, **`platform.core.create_visualization`**, or the step 6 last-seen **`platform.core.execute_esql`** in the **same** tool round. Finish **attachment + one validation `execute_esql`** first; then run complementary tools in following round(s). **Exactly one** **`observability.remediation_workflow`** call is **mandatory** when this skill applies: run it **alone** in its dedicated round (ladder step 5), after all other validation / complementary work. **Step 6 (post-remediation verification)** is the **terminal** phase — its **`platform.core.create_visualization`** and last-seen **`platform.core.execute_esql`** calls each run **alone** in their own tool round, **never batched** with `observability.remediation_workflow`, `ki_search`, complementary expansion tools, or each other.

### `ki_search` — narrow and late

**Do not call `ki_search`** when **all** of the following are true:

- You already ran validation ES|QL and complementary expansion as required above, and
- You have enough to answer remediation without indicators, and
- The user did not ask for KIs or alerts.

**Call `ki_search` only when** the user asks for indicators/alerts, or you need **one** narrow retrieval tied to **`rule_names`** + **`cause_kis[].name`** after expansion. Use **low `limit`**, **`stream_names`**, minimal `search_text`.

### APM / services (`get_services`, `get_service_topology`, `get_traces`)

When **`dependency_edges` is non-empty** and validation shows **still active** (or the user asked blast radius / caller impact), **`get_services`** for `source`/`target` is **required** as part of complementary expansion — not optional. **`get_service_topology`** / **`get_traces`** remain optional under the caps and empty-topology rule in “Complementary expansion” above.

### `get_logs`, `generate_esql`, `run_log_rate_analysis`

- **Required** when a **`recommendations[]`** line clearly implies a log or rate query that step 2 did not cover.
- **Otherwise** use for contradictions or thin validation — cap **extra** ad-hoc ES|QL beyond step 2 to **two** additional **distinct** pipelines total (including the “second rule” bullet in complementary expansion).

### Non-duplication

Never rerun the **same** `evidences[].esql_query` string or a semantically identical pipeline to “be thorough.” If you need volume or trend, use **`STATS` / buckets** or a **different filter** so the result is not the same few rows the attachment already summarized.

### Scope discipline in the answer

Do not chase unrelated KI or log themes unless tied to this event’s **`rule_names`**, **`cause_kis`**, **`dependency_edges`**, **`recommendations[]`**, or **`root_cause`**.

## Synthesis

- **Cause story**: Start from **`root_cause`**, then add what validation and **complementary** tools changed or proved.
- **Remediation**: Start from **`recommendations[]`**; cite new evidence from expansion where it supports or reprioritizes steps; mark human / out-of-band items when needed.
- **Remediation workflow (required)**: Ladder step 5 **always** runs **`observability.remediation_workflow`** once — incorporate its result (workflow execution status / error) and the **`service_name`**, **`namespace`**, and rationale reflected in your **`query`** into your closing remediation summary. Also report the **plan** that was approved and the workflow's **closure message** echoing that plan, so the user sees what was simulated.
- **Post-remediation verification (required)**: Ladder step 6 **always** produces the area chart from **`platform.core.create_visualization`** and a single last-seen `@timestamp` (or "no matches in window") from **`platform.core.execute_esql`**. The closing user-facing message **must** explicitly surface all three — (a) the **workflow execution status** from step 5, (b) the **area chart** showing the post-remediation trend, and (c) the **last-seen `@timestamp`** — and use them to state whether anything has changed since the remediation action.

---

## Anti-patterns

- Do not run broad **`get_logs`** before the deduped evidence validation **`execute_esql`** when that query exists.
- Do not ask which stream to use when `stream_names` is populated.
- Do not fabricate results; tie claims to the attachment or tool output you ran.
- Do not end the conversation on the remediation workflow round; you **must** still complete **step 6** (post-remediation verification) afterwards.
- Do not end the conversation without calling **exactly one** **`observability.remediation_workflow`** (step 5) **and** the **step 6** post-remediation verification — **`platform.core.create_visualization`** plus a last-seen **`platform.core.execute_esql`** — as the **final** tool rounds for this skill.
- Do not paste the promoted evidence ES|QL **verbatim** into **`platform.core.create_visualization`**; adapt it into a time-bucketed aggregation. Do not run both last-seen variants (adapted `STATS MAX(@timestamp)` and a narrow rerun) — pick one.
- Do not switch to a different evidence row in step 6 than the one validated in step 2.
- Do not call **`observability.remediation_workflow`** without including a remediation plan derived from **`recommendations[]`** in the **`query`** (and prioritizing any rollback / revert / redeploy-previous step as step 1).

---

## Tips

- Pass **`time_range`** to **`execute_esql`** / **`generate_esql`** when supported.
- Prefer aggregates and tight `LIMIT` on row-returning ES|QL.
- If ES|QL errors on field names, adapt minimally and retry **once**.
