# Domain: Rule Management

## What every reviewer must know

This domain covers **regular management operations on detection rules**: CRUD, enable/disable, bulk actions, import/export, history, search/find — including when those operations target prebuilt rules. It explicitly does NOT cover the prebuilt-rules install/upgrade/customization/deprecation workflow, rule monitoring, rule execution, or detection RBAC — those are sibling domains with their own knowledge captures (prebuilt-rules customization has actively leaked into rule_management code; consolidation is under consideration).

The architecture is layered: API contracts in `common/api/detection_engine/{model/rule_schema,rule_management}/**` (Zod-generated route schemas) → server route handlers in `server/lib/detection_engine/rule_management/api/**` → a single facade `IDetectionRulesClient` (`server/.../logic/detection_rules_client/`) that wraps the Alerting Framework's `RulesClient` and dispatches to per-method modules under `methods/`. Every detection-rule operation MUST go through AF's `RulesClient`; every Saved Object MUST go through `SavedObjectsClient`. `IDetectionRulesClient` is the abstraction boundary — business logic lives on it, not in route handlers and not in `utils/`.

The primary domain entity is "detection rule" with three distinct models that you cross between: `RuleResponse` / `RuleCreateProps` / `RuleUpdateProps` / `RulePatchProps` (API contract, snake_case, the single source of truth for what fields are public) — backend rule-params schema (per rule type, in `server/.../rule_schema/model/rule_schemas.ts`) — `RuleAlertType = SanitizedRule<RuleParams>` (what AF's `RulesClient` returns). On writes, everything passes through `convertRuleResponseToAlertingRule(...)` before `RulesClient.<op>`; on reads, everything passes back through `convertAlertingRuleToRuleResponse(...)` (or `internalRuleToAPIResponse` for the no-Zod escape hatch). API contracts must NEVER leak SO/AF storage shape (e.g. camelCase `alertTypeId`, `createdBy`, `lastRun`); they expose only fields defined by `RuleResponse` in snake_case.

API contracts must be designed for **longevity**: while versioning is technically supported, no public Kibana endpoint has ever versioned in practice, there are no guidelines, and the API-docs generator does not support it well. Treat the contract as frozen once shipped — design for evolutionary extensibility (objects over flat fields, arrays of objects over `string[]`) up front. **MSSP scale is real**: the largest customers run 300 Kibana spaces per cluster with 2000–10000 rules per space (up to ~3M rules per tenant). New bulk endpoints must do **true bulk** under the hood; promise-pool / p-map / async-loop batching do not scale at that size.

Historical naming legacy: in code, "rule" is sometimes called "alert" — the rule's saved-object type is literally `alert`. In current product language, "alert" means a **detection alert** (the document a rule produces). Don't try to rename; learn to read past it.

## Architectural invariants

- **Never bypass the abstraction.** Detection rules are read/written **only** through AF's `RulesClient`. Own saved objects are read/written **only** through `SavedObjectsClient`. Direct ES access is rare and must be justified.
- **Business logic lives in clients, never in route handlers or `utils/`.** `IDetectionRulesClient` is the rule-management abstraction (other clients exist for adjacent concerns). Two named anti-patterns to refuse on review: the *utils anti-pattern* (logic scattered in random small `utils/*.ts`) and the *thick API endpoint handler anti-pattern* (logic stuffed inside `route.ts`). Route handlers validate, authorize, call **one** client method, and shape the response — nothing else.
- **API endpoints accept and return domain-model entities, never raw data shapes.** Never expose SO / AF / ES storage shape. The API contract is defined by `RuleResponse` (and its companions `RuleCreateProps` / `RuleUpdateProps` / `RulePatchProps`); fields are snake_case; field names that match `RuleResponse` are the canonical names.
- **Cross the model boundary through the canonical converters.** Inbound (API → AF): `convertRuleResponseToAlertingRule(ruleWithUpdates, actionsClient)` — must run before any `RulesClient.<write op>`, with EVERYTHING (rule params, actions, every nested field) converted to AF's model. The authoritative reference is `server/.../detection_rules_client/converters/convert_rule_response_to_alerting_rule.ts` — the code is the documentation. Outbound (AF → API): `convertAlertingRuleToRuleResponse(...)` (with Zod validation) or `internalRuleToAPIResponse(...)` (without Zod — escape hatch). Other helpers exist (`internalRuleToAPIResponse`, `convertPrebuiltRuleAssetToRuleResponse`, "mergers" under `logic/detection_rules_client/mergers`); know they exist and don't write a fourth one without a clear reason.
- **ES-derived data must be normalized before it leaves the endpoint.** Simple normalization (case + sane defaults) lives in `converters/normalize_rule_fields_common.ts`. Complex normalization (cross-field, deprecated-field handling, AF→API translation) lives across `converters/normalize_rule_source.ts`, `common/detection_engine/transform_actions.ts`, `rule_management/normalization/rule_actions.ts`. Skipping normalization leaks AF-shape (most commonly action parameters) into responses.
- **Design APIs for longevity, not versioning.** Group related parameters into nested objects (e.g. `gaps: { fill_statuses, range_start, range_end, scheduler_id }`), not flat siblings. Use arrays of objects (e.g. `counts: [{ category: 'tags', size: 1000 }]`) over `string[]` so each entry can be parameterized later. Model `sort` as `Array<{ field, order }>` (max size 1 for now) so multi-field sort is non-breaking. Don't import legacy notation from older endpoints into new endpoints — if `FindRulesSortField` carries `createdAt` / `riskScore` / `updatedAt` "for backwards compatibility", split the enum or accept only canonical names on the new endpoint.
- **Type aggregation results precisely; no `Record<string, unknown>` + `as` casts.** Use `aggregations?: Record<string, AggregationsAggregate>` from `@elastic/elasticsearch` and parse defensively. `as`-casts in cross-layer code hide shape mismatches until runtime (see Thread 26 of #262307 — `lastRunStatus` aggregation returned a 400 because the field path was wrong; type narrowness would have surfaced it).
- **Diff algorithm: array fields change at the array level, not per-element.** For fields like MITRE ATT&CK threats — arrays whose elements reorder, get added/removed, or update in place — the diff records "the whole array changed". This is RFC 7396 (JSON Merge Patch) semantics with a domain twist: rule history is consumed by humans who want "MITRE techniques changed", not 5 reorder events. Enforced by a unit test on `compute_old_values.ts`.
- **REST URLs: nested resources use `/{parent_id}/sub_resource`.** Rule history is `GET /api/detection_engine/rules/{ruleId}/history`, not a flat `/rules/_history`. Document the URL shape in the API design proposal before opening the PR.
- **Heavy endpoints must rate-limit, time-out, paginate, and minimize response payload.** Concrete patterns to copy:
  - Rate limiting / concurrency: `tags: [routeLimitedConcurrencyTag(MAX_ROUTE_CONCURRENCY)]` — see `server/.../api/rules/bulk_actions/route.ts`.
  - Socket timeout: `idleSocket: RULE_MANAGEMENT_IMPORT_EXPORT_SOCKET_TIMEOUT_MS` — see `server/.../api/rules/import_rules/route.ts`.
  - Pagination: `find_rules/route.ts`, `search_rules/route.ts`.
  - Minimal/aggregated payload: `bulk_actions/route.ts`.
- **Bulk endpoints must do true bulk under the hood.** New bulk endpoints must call true-bulk primitives (`RulesClient.bulkEdit`, etc.), not loop with `p-map` / promise-pool / async-batching. The only true bulk endpoint today is `bulk_actions/route.ts` (action edits via `RulesClient.bulkEdit`); every other bulk endpoint processes rules one-by-one in some batch mode and is known not to scale to MSSP customers.

## Common review patterns (learned from real PRs)

- **Keep route handlers linear; don't pre-extract helpers when extraction adds complexity.** When a helper just shuffles state between itself and the route and returns a discriminated union the route re-destructures, prefer inlining the logic behind an `if (...) { … }` block. Extract only when the helper has a coherent contract independent of the route. (PR #262307 Threads 20, 22 — @dplumlee.)
- **Parallel folder structure in `common/api/.../rule_management/`.** New contract files (`*_route.gen.ts`, `*_route.schema.yaml`, related types) go into a per-feature subfolder (e.g. `granular_rules/`, `model/search/`) — never at the top level next to CRUD folders. (PR #262307 Threads 17, 44 — @dplumlee, @banderror.)
- **Validate cross-field invariants in request schemas.** If start/end date pairs (or similar) can be inverted, reject with 400 in the request-schema validator — don't silently return an empty result. (PR #262307 Thread 21 — @dplumlee. Accepted as future work in that PR.)
- **Use real booleans in conditional predicates, not truthy/falsy strings.** `gapFillStatuses && gapFillStatuses.length > 0 && gapsRangeStart && gapsRangeEnd` infers as `string | boolean | undefined`; prefer `(gapFillStatuses?.length ?? 0) > 0 && Boolean(gapsRangeStart) && Boolean(gapsRangeEnd)`. (PR #262307 Thread 37 — @dplumlee + @banderror, cross-reviewer reinforcement.)
- **Prefer in-tree implementations over new npm dependencies for small algorithms.** Especially when the dep would have to clear Kibana's security review. Promote to a `@kbn/*` package later if reused. (PR #268724 Thread 7 — @sdesalas/@maximpn on the JSON-merge-patch implementation for rule history.)
- **AI-assisted review is the team's normal practice in this area.** `dex-review-pr` (role-based AI reviewers — architect, clean-code, etc.) is routinely run on PRs in this domain. The human reviewer's job is to pick out the architecturally load-bearing items from the ~20–30 AI-surfaced findings and combine them with the architectural concerns only a domain expert sees. Treat AI-surfaced concerns the same as human reviewer concerns. (See PR #262307 review comments where AI-surfaced issues are credited directly.)

## Security considerations

- **API Key inheritance is the headline concern.** The user who **created or last updated** a rule (with a few edge cases — enable/disable do NOT refresh the key) gives the rule an API Key with that user's privileges. The rule then queries source data with that user's scope. Admin-level users typically have access to multiple Kibana spaces, so the rule starts querying source events across spaces, not only the one it's scheduled in. There is no native "kibana spaces for source data" primitive, but enterprise customers set up multi-tenancy manually by namespacing source data per space — this behavior **breaks their RBAC-based multi-tenancy**. Reviewers of any change that touches rule create / update / patch / import / bulk-edit must consider whether the change affects when the API Key is refreshed and which user provides it.
- **Sensitive-looking content in rule fields is by design.** Rule queries, investigation guides, notes, and `meta` legitimately contain IPs, hostnames, asset names, and other customer-internal text. There is **no obfuscation** of these fields in the rule-management API or UI, and there should not be. Don't add scrubbing logic in this domain — telemetry scrubbing is the responsibility of the telemetry domain.
- **RBAC / field-level authz is out of scope of this domain file.** The 9-flag `DetectionRulesAuthz` shape (`canEditRules`, `canEnableDisableRules`, `canManualRunRules`, `canEditExceptions`, `canEditCustomHighlightedFields`, `canEditInvestigationGuides`, …) is governed by a separate subdomain with its own knowledge capture. Adding a new field-level capability flag is **not a simple pattern-match** — escalate to that subdomain's experts.
- **Open / under-researched:** NDJSON import attack surface beyond Zod validation (action connector reference resolution in the current space, exception-list ownership checks, prebuilt-rule asset reference spoofing) needs investigation. License/tier feature matrix (e.g. ML rules on Basic) needs separate research.

## Performance constraints

- **Hot paths, ranked by impact:**
  1. `server/.../api/rules/bulk_actions/route.ts` — by far the heaviest endpoint in the area.
  2. `server/.../api/rules/import_rules/route.ts`.
  3. `server/.../api/rules/export_rules/route.ts`.
  4. `server/.../api/rules/search_rules/route.ts` — driven by aggregation support.
- **Scale to assume**: 300 Kibana spaces per cluster × 2000–10000 rules per space at the largest MSSP customers (worst case ~3M rules per tenant). Any change that scales with rule count or space count needs explicit consideration of this regime.
- **`search_rules` is called every 60 seconds** by the Rule Management table auto-refresh. No other endpoint is polled at unusual frequency.
- **Known N+1 traps inside AF's `RulesClient.bulkEnable` and `bulkDisable`.** These are upstream of this domain but rule_management is the heaviest caller. Don't add new callers to these methods naively; surface the N+1 if it bites you.
- **True-bulk discipline.** Today only `bulk_actions/route.ts` is a true-bulk endpoint (via `RulesClient.bulkEdit` for action edits). Every other bulk endpoint loops one-by-one in promise-pool / p-map / async-batching mode and is known not to scale to MSSP customers. New bulk endpoints must use a true-bulk primitive under the hood; if the primitive doesn't exist in AF yet, that's the work — don't ship a batched-loop version "for now".
- **Memory pressure on import and bulk-edit.** Regular rule-management is lighter than prebuilt-rules (which has ~1800 rules processed via expensive ES aggregations + server-memory processing), but the same memory considerations absolutely apply to `import_rules` and `bulk_actions` endpoints. Batch sizes and intermediate-array allocations are review hot spots here.
- **Aggregation perf at scale is an open question** for the new `_search` endpoint. A perf testing report exists and is awaiting team review — reference it before optimizing.
- **Canonical roadmap for scale/perf work:** https://github.com/elastic/security-team/issues/16958.

## Historical catches

- **PR #262307 ([Security Solution] Add rules/_search endpoint)** — multiple human reviewers approved the PR. The reviewer (@banderror) ran `dex-review-pr` which surfaced ~25 issues, then combined the AI-generated output with their own architectural concerns. The architecturally load-bearing items — **data-model leakage** (camelCase SO/AF attribute names exported as a `common/` API enum that PR #266112 will inherit across multiple endpoints) and **abstraction-boundary violation** (gap-filtering business logic split between `route.ts` and `utils/utils.ts` instead of behind a new `IDetectionRulesClient.searchRules` method) — would have been missed by a generic reviewer because the enum looked syntactically fine without knowledge of the upcoming contract freeze, the route structure looked clean without knowledge that `IDetectionRulesClient` is *the* abstraction in this domain, and reusing `FindRulesSortField` looked prudent without knowledge that the enum carries explicit legacy aliases meant only for backwards compatibility on older endpoints.
- **PR #268724 ([Security Solution] Implement Rule Changes History API)** — @sdesalas (the original author of the diff/merge-patch algorithm) caught that the unit tests did not cover the **array-as-unit diff semantics** that MITRE ATT&CK fields rely on. The proposed test would have passed and the diff would have produced misleading per-element change records when rules' threats arrays were reordered. A generic reviewer would have missed it because RFC 7396 by itself does not enforce array-level granularity; the domain twist (humans consume rule history and want "MITRE techniques changed", not 5 reorder events) lives in product-language history, not in the algorithm code. @sdesalas insisted on the unit test before approving.
