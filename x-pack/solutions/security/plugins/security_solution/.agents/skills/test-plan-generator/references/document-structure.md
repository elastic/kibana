## Notes

- **Terminology** is optional — omit the section entirely if the issue uses no domain-specific terms.
- **RBAC / Permission Testing Matrix** — only include if the issue explicitly mentions roles, permissions, or access control. Do not add speculatively.
- If a section cannot be filled in due to missing information, apply the Core rule: stop and ask the user rather than leaving placeholder text in the draft.

## Pending work pattern

When the issue passed to the skill is an epic and one or more sub-issues are not yet implemented, document the not-yet-implemented work as forward-looking scenarios using this pattern:

1. **Acceptance Criteria**: prefix each unimplemented AC with `(Pending #N)` where N is the sub-issue number — e.g. `25. (Pending #16937) Inline UDF definitions are translated correctly...`.
2. **Test Scenarios**: group the corresponding scenarios in a single collapsible section titled `Pending work — forward-looking gaps from open sub-issues`. Each scenario title ends with `— pending #N`.
3. **Test Coverage Summary**: add a single `Pending work` row whose `Automated` column reads `0` and `Manual only` equals the total — these scenarios are not testable today.
4. **Known Limitations**: add one ⚠️ entry noting that forward-looking scenarios are written against sub-issue ACs and should be reviewed when each sub-issue ships.

Apply this pattern only to **unimplemented** sub-issues. Implemented-but-unreleased sub-issues count as in scope — write regular scenarios for them. If the issue is not an epic at all, omit the pattern entirely.

## Test Coverage Summary — filling in the table

The `Automated` and `Manual only` columns are derived from the test coverage catalog built in Step 1:

- **Automated** — count the scenarios in this area that have at least one matching test (unit, integration, API integration, or e2e) in the test coverage catalog. In parentheses, note the test type(s) present — e.g. `2 (unit)`, `3 (unit, e2e)`, `1 (unit, partial)`. Use `partial` when a test exists but covers only a subset of the scenario's behaviour.
- **Manual only** — count the scenarios in this area with no matching tests at all (`No existing tests found covering this scenario`). These are the scenarios that require manual execution and represent the testing gap.
- The **Total** row must sum each column correctly. Double-check before saving.
- **Automated coverage notes** must include: the test framework and directory path, the specific key test files, and a plain statement of what is not yet covered (no e2e, no integration, etc.). Do not leave this as a generic placeholder — derive it from the test coverage catalog.

## Issue Clarity Assessment section

This collapsible section is the final block of every test plan, placed between *Test Execution Notes* and the footer. It is included **in every published test plan**, whether the scores are high or low — its purpose is to surface signal about issue authoring quality to PMs and writers as part of the published artifact.

The full rubric, the per-issue procedure, the combined-readability rules, and the Coverage Ratio computation are defined in [`issue-clarity-assessment.md`](issue-clarity-assessment.md). The canonical markdown layout for the section is defined in [`output-formats.md`](output-formats.md#issue-clarity-assessment-section). The block is wrapped in `<details>` so it appears collapsed by default in the GitHub comment, keeping the test plan readable while still giving the assessment first-class presence.
