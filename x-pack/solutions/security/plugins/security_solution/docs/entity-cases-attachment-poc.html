Entity Analytics — Cases attachment POC

A walkthrough of the feat/entity-analytics-case-attachments diff, which adds a new security.entity unified case attachment so entities (users, hosts, services, generic) can be attached to a case from Entity Analytics surfaces.

Overview

This branch is a proof of concept for the Entity Analytics → Cases bridge (OneCases). It teaches the cases plugin about a new attachment kind, security.entity, and wires an "Add to case" entry point into the entity flyout so an analyst can attach a user (eventually any entity) directly to a case.

The POC is gated behind a new entityAttachmentsEnabled experimental feature flag. With the flag off, the diff is inert: no UI appears, no attachment type is registered, and the cases plugin behaves exactly as before.

Scope. Today the only entry point is the user flyout header. The plumbing (schema, attachment definition, "Add to case" button component) is already type-agnostic, so adding host, service, and generic entry points is mostly wiring.

Source: x-pack/solutions/security/plugins/security_solution/common/cases/attachments/entity.ts

Architecture

Two plugins cooperate. The cases plugin (platform) gains a new attachment type id in its constants and unified-type sets. The security_solution plugin registers an attachment definition for that id on both the server (zod schema) and the client (display name, avatar, lazy-loaded children component) and exposes the "Add to case" UI.

Attachment payload

The on-the-wire shape is one zod schema that lives next to the constant, so the public registration, the server registration, and the runtime payload are all validated by the same definition.

// common/cases/attachments/entity.ts
export const ENTITY_ATTACHMENT_TYPES =
  ['user', 'host', 'service', 'generic'] as const;

const EntityAttachmentMetadataSchema = z.object({
  entityName: z.string(),
  entityType: z.enum(ENTITY_ATTACHMENT_TYPES),
  riskScore: z.number().optional(),
  riskLevel: z.string().optional(),
}).strict();

export const EntityAttachmentPayloadSchema = z.object({
  type: z.literal(SECURITY_ENTITY_ATTACHMENT_TYPE),
  owner: z.string(),
  attachmentId: z.string(),
  metadata: EntityAttachmentMetadataSchema,
}).strict();
Fields persisted with the attachment.
Field Type Required Notes
type 'security.entity' Yes Discriminator. Matches SECURITY_ENTITY_ATTACHMENT_TYPE.
owner string Yes Cases owner (e.g. securitySolution).
attachmentId string Yes Entity identifier (user name, host name, IP).
metadata.entityName string Yes Display label in the case timeline.
metadata.entityType 'user' | 'host' | 'service' | 'generic' Yes Mirrors the Entity Analytics types.
metadata.riskLevel string No Captured at attach time; rendered if present.
metadata.riskScore number No Captured at attach time; rounded for display.

Snapshot semantics. Risk fields are persisted with the attachment so the case timeline renders without re-fetching the entity. They reflect risk at attach time, not the live value.

User flow

End-to-end, attaching a user from the entity flyout looks like this.

  1. 1 The analyst opens the user flyout. UserPanelHeader reads the experimental flag with useIsExperimentalFeatureEnabled('entityAttachmentsEnabled') and, if on, renders EntityAddToCase next to the existing risk-level badge.
  2. 2 The analyst clicks Add to case. The popover menu shows Add to new case and Add to existing case. Both are disabled if the entity has no name or the user lacks createComment/update on cases (see useCaseDisabled).
  3. 3 Either action calls generateEntityAttachmentsMetadata + generateEntityAttachmentsWithoutOwner to build a CaseAttachmentsWithoutOwner array, then opens the cases plugin's flyout/modal with { attachments } or { getAttachments }.
  4. 4 The cases plugin posts the attachment, validating the payload against EntityAttachmentPayloadSchema on the server.
  5. 5 When the case is later viewed, the unified framework looks up the security.entity definition, lazy-loads AttachmentChildren, and renders the persisted metadata via EntityChildren.

Enabling the POC

Add the flag to kibana.dev.yml:

xpack.securitySolution.enableExperimental:
  - entityAttachmentsEnabled

What turning the flag on does:

Server

Server attachment registration

registerCaseAttachments calls attachmentFramework.registerUnified(getEntityAttachmentType()), so case API requests carrying a security.entity attachment are validated against the shared zod schema.

Public

Public attachment registration

The setup phase of the security_solution public plugin registers getEntityAttachment(), so the cases UI knows how to render an security.entity attachment in the timeline.

UI

Flyout entry point

UserPanelHeader renders the Add to case button beside the risk badge in the user flyout. With the flag off, the button is not rendered at all.

Cases plugin defaults. The new constant in UNIFIED_ATTACHMENT_TYPES and MIGRATED_ATTACHMENT_TYPES is shipped unconditionally (the cases plugin has no flag of its own here). Only the security_solution registrations and UI are gated. That is fine because the cases plugin only persists/renders the attachment when something is registered for the id, but it is worth knowing.

Notes and follow-ups

Why a "unified" attachment

The cases plugin distinguishes legacy v1 attachments from the new unified-only ones. security.entity is unified-only by design, which is why it is in UNIFIED_ATTACHMENT_TYPES and MIGRATED_ATTACHMENT_TYPES but absent from PERSISTABLE_STATE_LEGACY_TO_UNIFIED_MAP. The inline comment in attachments.ts spells this out.

Why a snapshot of risk

EntityChildren renders straight from metadata, so the timeline is fast and offline-rendering safe. The trade-off is that risk in the case will diverge from the live entity store value. A future iteration can fetch live values when the attachment opens, but that is out of scope for the POC.

Adding more entry points

EntityAddToCase is type-agnostic — it takes an EntityToAttach with an id, name, type, and optional risk fields. To add the host or service flyout, drop the same component into the corresponding header and pass the right entity shape. No changes required in the cases plugin or the schema.

Open questions

  • Should host/service flyouts also surface the button, or is user-only enough for the first iteration?
  • Do we want a dedicated "Entity" section in the case detail view, or is the unified timeline rendering sufficient?
  • Does the case need to reflect risk drift, and if so, on read or via a background reconcile?
