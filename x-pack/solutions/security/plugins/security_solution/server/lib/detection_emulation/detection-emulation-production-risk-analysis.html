Production Risk Analysis

Detection Emulation Skill

Risk surface and mitigation roadmap for the Agent Builder skill that lets an LLM dispatch attack-emulation actions against enrolled endpoints.

Audience: Product Management Scope: x-pack/solutions/security/.../detection_emulation Status: Production-opt-in (both feature flags ship dark by default) Last updated: 2026-05-14

1TL;DR for the PM

Current posture

The skill is safe to flip on for opt-in beta with an operator-configured allowlist. Both feature flags ship dark by default; enabling them requires explicit operator action. There are no Critical or High residual risks; the remaining items are documented Medium/Low entries with named scheduled fixes or permanent-constraint paragraphs (cost + fallback + monitoring trigger).

Active safeguards

Architecture. The skill exposes six tools to the LLM: validateRule, getEmulationHistory, and four typed per-family tools — runProcessCommand, runFileCommand, runNetworkCommand, runExecutionCommand. The four per-family tools share a withCommandGates helper that enforces feature flag → auth → RBAC → schema fanout cap → default-deny allowlist → per-space rate → per-host rate → concurrency gate → mandatory HITL → audit attribution. The free-form execute.parameters.command surface mirrors the Kibana Console substrate; every dispatch is bounded by the gate sequence above.

Residual risk is concentrated in ≤1 Medium item (register row #4 — reduces to Low when allowedExecuteCommandPatterns is configured) and 4 Low items: #3 (prompt-injection/HITL residual), #13 (error-oracle permanent constraint), #14 (upload command, mitigated), #16 (kibana_system index-read residual, documented in README). Two permanent constraints remain — R-N1 in-memory single-process gates and R-N2 unsigned ActorContext. Every item has a named fix path, a permanent-constraint paragraph with cost + fallback + monitoring trigger, or both. See the roadmap section for the full list.

2What the LLM can do

The skill is gated on a two-tier opt-in:

  • Tier 1 — read-only / synthetic (experimentalFeatures.detectionEmulationLogInjection = true): the LLM gets validateRule in log_injection mode and getEmulationHistory. No live endpoint actions; no allowlist required because no host is touched. Synthetic ECS docs flow into a space-scoped, ILM-managed index pattern.
  • Tier 2 — live endpoint actions (experimentalFeatures.detectionEmulationRealExecution = true AND runtime kill switch detectionEmulation.realExecutionEnabled = true): the four run*Command per-family tools become callable, and validateRule can be invoked in real_execution mode. Both require the calling user to hold Endpoint response-action privileges and the operator to have explicitly added the target host to xpack.securitySolution.detectionEmulation.allowlist.endpointIds.

Tier 2 lets operators stage a rollout — turn on log_injection for unattended scheduled validation today, leave real_execution off until the operator is ready to underwrite an allowlist + per-host budget. The complete tool surface is six tools:

  • validateRule — runs a curated payload against a rule (real EDR or log injection) with mandatory HITL for real_execution
  • getEmulationHistory — read-only history query
  • runProcessCommand — kill-process, suspend-process, running-processes (declarative HITL prompt)
  • runFileCommand — upload, get-file (declarative HITL prompt)
  • runNetworkCommand — isolate, unisolate (declarative HITL prompt)
  • runExecutionCommand — execute, runscript, memory-dump, scan (declarative HITL prompt)

Every per-family tool runs through a shared withCommandGates helper that enforces:

  1. Feature flag (detectionEmulationRealExecution) AND runtime kill switch (realExecutionEnabled)
  2. Authentication
  3. RBAC (per-command, delegated to substrate's RESPONSE_CONSOLE_ACTION_COMMANDS_TO_REQUIRED_AUTHZ)
  4. Schema fanout cap (MAX_ENDPOINT_FANOUT = 5)
  5. Default-deny allowlist (allowAll: false)
  6. Per-space rate limit (100 / space / hour)
  7. Per-host rate limit (3 / host / hour)
  8. Concurrency gate (≤1 real_execution in flight per space)
  9. Mandatory HITL prompt (Agent Builder framework primitive, declarative on per-family tools, on-demand on validateRule)
  10. Audit attribution with actor.kind discriminator
Note on execute.parameters.command

The execute.parameters.command string is free-form on the substrate side (it matches the Kibana Console surface). The bound is defense-in-depth: the per-family tool split + HITL + RBAC + default-deny allowlist + per-host rate + fanout cap + concurrency gate. The LLM cannot issue a free-form execute without the user clicking through a HITL prompt that shows the command, endpoints, and emulation ID. See Risk #4 below.

Nine gates protect the path from prompt to host. Gate 1 is two-keyed: the static feature flag experimentalFeatures.detectionEmulationRealExecution AND the runtime kill switch detectionEmulation.realExecutionEnabled. 5Risk deep dive

Detail on the items most relevant to the production-opt-in decision. Cards are color-coded by status (green = closed, blue = mitigated, gray = permanent constraint).

RISK #1 · Closed

Allowlist is default-deny on every entry point

execution/allowlist.ts · load_allowlist_config_from_operator_config.ts

createDefaultAllowlistConfig() returns { allowAll: false, allowedHosts: new Set() }. Both REST routes (api/run_command/route.ts, api/validate_rule/route.ts) and all five tool surfaces (the four per-family tools + validateRule) read config.detectionEmulation.allowlist via the same helper (loadAllowlistConfigFromOperatorConfig), so an operator who configures xpack.securitySolution.detectionEmulation.allowlist.endpointIds: [host-a, host-b] has the same allowlist enforced everywhere. If an operator explicitly sets allowAll: true, the plugin logs a warning at bootstrap.

Tests: api/run_command/route.test.ts, api/validate_rule/route.test.ts, plus per-tool jest specs all cover "operator config drives allowlist" and "default-deny rejects every host".

RISK #2 · Closed

Human-in-the-loop confirmation is mandatory

build_emulation_confirmation.ts · per-family tools · validate_rule_tool.ts

Each of the four per-family tools declares confirmation: { askUser: 'once', getConfirmation }. The Agent Builder framework (not the skill or the LLM) renders a HITL prompt with the command, endpoint IDs, emulation ID, and parameters; the user must click "Confirm" or "Reject". The LLM cannot bypass — the framework intercepts the tool call.

validateRule uses the on-demand HITL primitive: checkConfirmationStatus → if unprompted, return askForConfirmation; if rejected, return user_declined; only proceed past the gate sequence when accepted. Skipped only in executionMode === 'standalone' (eval / A2A friction-free runs).

RISK #3 · Mitigated

Prompt injection → real command execution

All LLM-readable surfaces (alerts, cases, hunting results, rule descriptions)

An end-to-end RCE chain via prompt injection requires all of the following to coincide:

  1. Attacker text is read by the LLM
  2. LLM picks one of the four per-family tools (the LLM has to commit to a family)
  3. User clicks "Confirm" on the framework-rendered HITL prompt that shows the command, endpoints, and emulation ID
  4. Target host is on the operator-configured allowlist (default-deny)
  5. ≤5 endpoints in the call
  6. Per-host bucket has capacity (3/host/hour)
  7. No other real_execution scenario in flight
  8. Calling user holds the per-command RBAC privilege

Each link is independently necessary; the user's HITL approval is the load-bearing one. The residual is the case where a user is socially engineered to approve a confirmation prompt that includes attacker-supplied content.

RISK #4 · Permanent constraint

execute.parameters.command is a free-form string

runner.ts · substrate matches Kibana Console execute surface

Why permanent: the substrate's ResponseActionsClient.execute method is the same surface used by the Kibana Console. Adding a Kibana-side shell-grammar parser or shell-string allow-list would diverge the two surfaces and silently break Console use cases. The substrate change would also have to land independently of detection_emulation.

Layered mitigation: the per-family tool split means the LLM has to pick runExecutionCommand explicitly; the HITL prompt then shows the exact command string to the user before dispatch; the default-deny allowlist means the operator has to opt-in each target host; the per-host rate limit caps damage to 3 commands per host per hour; the concurrency gate prevents parallel fan-out.

Cost: a user who is socially engineered to approve a HITL prompt with a malicious command can dispatch up to 3 commands per host per hour to up to 5 host(s) on the operator's allowlist. Fallback: operator can disable the detectionEmulationRealExecution feature flag at any time (no restart on flag flip; takes effect on next request). Monitoring trigger: any actor.kind = 'agent-builder' + command = 'execute' entry in the response-action audit log without a corresponding Agent Builder conversation ID is a signal the audit pipeline broke or someone is forging actor context — alert the SOC team.

RISK #5 · Closed

Audit trail records the LLM conversation

execution/audit_logger.ts · execution/audit_context.ts · emulation_report_type.ts

ActorContext is a discriminated union with kind: 'user' | 'agent-builder'. The agent-builder variant records:

  • conversationId
  • runId
  • toolCallId
  • promptHash (SHA-256 of the user prompt that triggered the LLM tool call)

buildEmulationComment appends a structured via=agent-builder/conv:<id>/run:<id>/tool:<id> suffix to the response-action comment. The detection-emulation-report SO carries an actor field on model version 2; legacy v1 rows are backfilled to actor: { kind: 'user' } on first read.

RISK #6 · Closed

Per-space rate limit is not the only ceiling

execution/rate_limiter.ts · execution/concurrency_gate.ts · common/.../constants.ts

Three layered caps bound an LLM run-amok: ≤5 endpoints per call (schema-layer), ≤3 commands per host per hour, and ≤1 in-flight real_execution scenario per Kibana space. The 100-command-per-space-per-hour bucket sits on top.

Rate-limiter acquisition is atomic with rollback: a partial fanout never half-charges either bucket. The concurrency gate returns concurrency_exceeded with inflight_scenario_fingerprint + Retry-After so the LLM can back off gracefully.

RISK #7 · Closed

Tool path enforces allowlist config

execution/load_allowlist_config_from_operator_config.ts

Both REST and tool surfaces construct the allowlist via the same loadAllowlistConfigFromOperatorConfig helper. An operator-configured allowedHosts is enforced uniformly across every entry point.

RISK #14 · Mitigated

upload command exposed to the LLM

run_file_command_tool.ts

upload lives in the runFileCommand per-family tool. The LLM has to pick file explicitly; the HITL prompt shows the upload destination and content hash to the user; the operator-configured allowlist limits which hosts can receive uploads; the per-host rate caps to 3 file operations per host per hour. Operators who don't want upload at all can disable the executeFileOperations RBAC privilege at the role level (existing substrate behavior).

7Active safeguards

Twenty-three controls are in force on every real_execution dispatch. The first six bullets and the last five are the load-bearing additions for production opt-in; the rest are platform-level defenses inherited from the wider security_solution surface.

  • Default-deny allowlist on every entry point (REST + 5 tool surfaces) — operator must explicitly opt-in each target host. Bootstrap warning if allowAll: true is configured.
  • Endpoint fanout cap: MAX_ENDPOINT_FANOUT = 5 enforced at the Zod schema layer before any auth check. Single source of truth shared across all schemas and the LLM-facing prose.
  • Per-host rate limit: 3 commands / host / hour. Atomic acquire-with-rollback so a partial fanout never half-charges either bucket. blockedEndpoints returned to the LLM for graceful retry.
  • Concurrency gate: ≤1 in-flight real_execution scenario per Kibana space. Stale-entry sweeper for crash recovery. Retry-After + inflight_scenario_fingerprint on rejection.
  • Runtime kill switch: realExecutionEnabled defaults to true; operators flip via kibana.yml reload (no restart) to halt new real_execution dispatches. Error likely_cause reports runtime_kill_switch_engaged so operators flip the right knob — not restart Kibana.
  • Mandatory HITL prompt on every destructive tool call (declarative on the four per-family tools, on-demand on validateRule). Framework-rendered; LLM cannot bypass.
  • Both modes ship dark behind explicit feature flags (detectionEmulationLogInjection, detectionEmulationRealExecution) — neither is on by default.
  • Authentication is mandatory; unauthenticated callers get 401 (no username='unknown' fallback).
  • Per-command RBAC delegated to the existing RESPONSE_CONSOLE_ACTION_COMMANDS_TO_REQUIRED_AUTHZ map — same enforcement as the Console.
  • Atomic rate-limiter acquire/release (no check-then-act race).
  • Idempotency cache absorbs LLM replays / network retries (caps duplicate dispatch).
  • Internal errors are not echoed to the LLM client — generic message + server-side detail.
  • Saved-object audit trail is space-scoped, hidden, write-once, with (scenarioFingerprint, ruleId) deduplication; actor block carries kind: 'user' | 'agent-builder' with conversation/run/tool IDs and SHA-256 prompt hash on agent-builder dispatches.
  • Wall-budget hard ceiling (5 min) on telemetry polling — no runaway long-poll connections.
  • Allowlist returns all blocked endpoints, not just the first — operators can remediate in one shot.
  • Single guardrail bundle per Kibana node: createDetectionEmulationGuardrails is invoked once in the plugin setup() path; the resulting { allowlist, rateLimiter, concurrencyGate, idempotencyCache } is dependency-injected into both REST routes and all five tool factories. No per-tool singletons; per-node budget = per-process budget.
  • agentType narrowed to z.literal('endpoint') in every tool schema. The RESPONSE_ACTION_AGENT_TYPE enum's other values (sentinel_one, crowdstrike, microsoft_defender_endpoint) are not wired through the runner; narrowing eliminates LLM dead-end paths and forces a deliberate schema change if a vendor expands.
  • Tool ID sanitization: every inline tool ID is run through sanitizeToolId before reaching the LLM connector layer; all six IDs assert round-trip safety + no post-sanitisation collisions in skills.test.ts. Closes the dot-in-tool-id Bedrock / Sonnet 4.5 routing failure class.
  • Curated-only mode (xpack.securitySolution.detectionEmulation.validation.curatedOnly): when enabled, Gate 1.5 in withCommandGates intersects the LLM's command/parameters tuple against the bundled payloads.json library — only fingerprints present in the library are accepted. Now covers execute (closes the previously unguarded path) and upload (closes the implicit short-circuit when parameters.command was absent). Default-off; intended for MSSP / multi-tenant deployments. Closes register rows #15, #4 (partial), #14 (partial).
  • Operator-tunable script allow-list (validation.allowedScriptIds): when populated, Gate 1.5 rejects runscript with any scriptId / script_id not on the operator-vetted list — defense-in-depth on top of the existing canWriteExecuteOperations RBAC privilege. Empty list = no-op. Closes register row #12.
  • Execute-command regex gate (validation.allowedExecuteCommandPatterns): operator-supplied array of JavaScript regex strings compiled at Kibana boot. Gate 1.5 rejects any execute call whose parameters.command matches none of the patterns; reason 'execute_command_not_allowed'. Empty array = no-op (existing behaviour). Reduces register row #4 from Medium to Low when configured.
  • ECS dataset stamp on synthetic docs (event.dataset: "emulation.synthetic", event.module: "emulation"): emitted on every log-injection document by log_injection/generator.ts; mapped as keyword fields in the index template. Custom rules that query broad index patterns can add an event.dataset != "emulation.synthetic" exclusion clause as a deterministic ECS-standard filter. Closes register row #9 (reduces from Medium to Low).
  • Operator-driven halt route (POST /internal/detection_engine/emulation/halt): pairs with the runtime kill switch — kill switch blocks new dispatches, halt invokes EmulationConcurrencyGate.cancelAllInflight() to release slots already in flight. Same canWriteExecuteOperations Endpoint privilege as the destructive commands. Closes register row #10 / R-N3.

8Open questions for the PM

  1. Who is the GA target customer? SOC engineers managing their own rules (low blast radius, high trust) versus MSSP-style multi-tenant deployments (higher blast radius, multi-customer leakage potential) imply different acceptable risk profiles. The current posture is safe enough for SOC-engineer opt-in beta; the runtime kill switch gives MSSPs a fast halt knob (and now the cancel-of-in-flight extension via the new halt route); curated-only mode (validation.curatedOnly) is also wired and ready for opt-in so multi-customer GA gates can be flipped piecewise.
  2. Is there appetite for a "sandbox endpoint" requirement? — i.e., the GA contract is "real_execution can only target endpoints tagged purpose: emulation-sandbox". The default-deny allowlist is operator-side enforcement of the same idea; sandbox tagging would just shift the allowlist from operator-configured to host-tagged. Decision: a soft contract (operator opts hosts in via allowlist) vs. a hard contract (only hosts with the tag are even allowable in the allowlist).
  3. What is the data-retention story? SO audit trail is currently retained indefinitely; emulation log index is 7 days. Does compliance need both at the same retention level? The actor attribution on the SO makes that trail forensically useful for arbitrarily long retention; the synthetic log index is intentionally short-lived and re-derivable from the SO trail.
