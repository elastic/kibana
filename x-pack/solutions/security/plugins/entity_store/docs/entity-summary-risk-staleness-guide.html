Entity AI summary: risk score staleness

How the flyout decides a saved summary is out of date, and how that interacts with history snapshot and the risk score maintainer.

Overview

When an analyst saves an AI summary on the entity flyout, Kibana stores a staleness snapshot on the entity document. On later visits, the UI compares live entity data to that snapshot.

For risk score, the rule is symmetric: if both the stored snapshot and the current entity.risk.calculated_score_norm are present, and they differ by more than 0.01, the summary is marked stale.

Source: entity_summary_staleness.ts — RISK_SCORE_EPSILON = 0.01

What is compared (defaults)

Default enabled staleness signals when a summary is saved
Signal Field on entity Stale when
risk_score entity.risk.calculated_score_norm Score changes by more than 0.01 (up or down)
rule_names entity.behaviors.rule_names New rule names only (removals ignored)
anomaly_job_ids entity.behaviors.anomaly_job_ids Not in defaults; same new-only logic if enabled

This guide focuses on risk_score, because it is the signal most likely to prompt regenerate during the history snapshot cycle.

Three background processes

  1. History snapshot (~24h): copies latest entities to a history index, then clears list-like entity.behaviors fields on latest. Does not change entity.risk.
  2. Log extraction: repopulates rule_names and anomaly_job_ids from logs when the entity appears in the extraction window.
  3. Risk score maintainer (~1h): recalculates scores from alerts, then may resetToZero entities that still had a positive score but were not updated in that run.

Risk maintainer run (simplified)

Hourly task (risk-score entity maintainer, ~1h interval)
  │
  ├─ Stage 1: Score base entities from alerts
  │     → new score + calculation_run_id for entities in this pass
  │     → bulkUpdate entity.risk on entity store (when dual-write enabled)
  │
  ├─ Stage 2: Resolution scoring (if applicable)
  │     → may update entity.relationships.resolution.risk
  │     → staleness compares entity.risk only, not resolution risk
  │
  └─ Stage 3: resetToZero (see next section)
        → entities still > 0 on disk but not rescored this run → 0

Staleness reads entity.risk.calculated_score_norm via buildEntitySummaryStalenessEntitySnapshot in the flyout.

Reset to zero — what and why

Reset to zero is stage 3 of the risk score maintainer. It is not the same as the history snapshot clearing entity.behaviors. History snapshot does not change entity.risk. Reset to zero changes risk when the engine decides the entity has no current risk inputs.

What it does

Each maintainer run creates a unique calculationRunId. Stages 1 and 2 score entities that have qualifying alerts (and resolution targets) and stamp their risk documents with that run id.

Stage 3 queries the risk score index and finds entities where:

  • the latest stored score is greater than 0, and
  • the latest calculation_run_id is not this run’s id (or is missing).

Those entities were not recalculated this hour but still look risky on disk. The maintainer writes score 0 to the risk index and, when id-based dual-write is on, to entity.risk.calculated_score_norm on the entity store.

Before reset runs, the risk index is refreshed so scores just written in stages 1–2 are visible. That avoids incorrectly zeroing entities that were scored moments earlier in the same run.

Up to 10,000 base and 10,000 resolution entities per run; remaining stale scores are reset on a later run.

Why it exists

Scoring is incremental per run. The maintainer does not recompute every entity every hour — only entities that appear in the alert-driven scoring passes.

Without reset to zero, an entity that had alerts last week would keep a positive risk score forever even when it has no contributing alerts in the current risk window.

Reset to zero is the cleanup rule:

If we did not recalculate you in this run and you still show score > 0, set you to 0. That means “no current risk inputs,” not “delete the entity.”

Risk engine setting (retain scores)

From Risk score preview UI copy (enableResetToZero)

When retain last calculated risk scores is selected, entities keep their last score until a new input (for example, an alert) triggers recalculation. When that option is off (default), scores reset to zero if no new inputs are found in the run.

Default configuration: enableResetToZero: true — scores decay to zero when not rescored.

Example: quiet host after a busy Monday

How risk_score on the entity document changes
When Event entity.risk.calculated_score_norm Summary snapshot (72.5)
Monday Alerts; maintainer scores host (run A) 72.5 —
Tuesday Analyst saves AI summary 72.5 Stored: 72.5
Wednesday No alerts in window; host not in stage 1 Still 72.5 (old run A on risk index) Still 72.5 — not stale yet
Wednesday resetToZero (run B) 0 Stale: “Risk score changed from 72.5 to 0”

The summary may still describe a high-risk host while the flyout now shows zero — prompting regenerate is appropriate.

Should entity-summary staleness treat reset differently?

No separate rule is required. Existing risk_score staleness already compares stored snapshot to current calculated_score_norm with a 0.01 epsilon, symmetrically (up or down).

Staleness vs maintainer outcomes
Outcome Stale for risk_score? Regenerate makes sense?
Small recalc (72.5 → 71) Yes Usually yes — posture shifted
resetToZero (72.5 → 0) Yes Yes — narrative likely outdated
Rescored this run, same score No No
enableResetToZero: false (score sticks) Only if a recalc changes the number Depends on retained score

If product later wants fewer nudges, tune epsilon, grace period after generated_at, or “stale only on increase” — that would also affect reset-to-zero drops unless you special-case zero.

Edge cases to know

  • Dual-write disabled — reset updates the risk index; entity store may not show 0, so flyout staleness may not see the change.
  • Resolution risk — reset can zero resolution scores; staleness only reads entity.risk, not entity.relationships.resolution.risk.
  • Batch limit — some entities reset on the next hour’s run; staleness may lag one cycle.
  • History snapshot — unrelated; it clears behavior lists only, not risk.

One-line mental model

Stage 1–2: "Who is risky right now based on current inputs?"
Stage 3:   "Who still looks risky on disk but we did not just recalculate? Set to 0."

Code: reset_to_zero.ts, risk_score_maintainer.ts (stage 3 after index refresh).

Flyout guard after in-session generate

Right after the user clicks Generate, the UI keeps a generation baseline in memory. Staleness is suppressed until live signals drift from that baseline. That avoids comparing against the old persisted snapshot before refetch completes.

After save + refetch, comparison is again: stored snapshot on the document vs current entity record.

Tuning options (product / engineering)

  • Larger epsilon — e.g. 1 or 5 points on the 0–100 scale; fewer nudges from small maintainer jitter.
  • Stale only on increase — ignore score drops and resetToZero; only prompt when risk rises.
  • Grace period — no risk staleness for N hours after generated_at.
  • Risk-only signals — drop rule_names from enabled_signals; simpler model, still subject to maintainer timing.
  • Server-side staleness API — flyout notes this should move server-side before GA for consistent behavior.

Reminder: list signals vs risk

If you keep new-only list staleness, the history snapshot clear does not falsely stale rule_names or anomaly_job_ids. The main regenerate pressure in that window is risk_score changing on the hourly maintainer while behaviors are temporarily empty.

If you later add removal-based list staleness plus “ignore when current is empty,” you mitigate history reset false positives but cannot distinguish “temporarily empty” from “genuinely quiet entity” without extra rules (time bounds, risk drop, etc.).
