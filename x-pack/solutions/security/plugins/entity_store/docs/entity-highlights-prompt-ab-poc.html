Entity highlights: old vs new prompt A/B POC

A breakdown of the dev-only prompt-variant switch on the highlights route, the matching UI toggle in the entity flyout, and the side-by-side comparison mode added to entity-summary-analysis.

Overview

The entity highlights flyout sends a system prompt to the LLM along with the entity summary. That prompt lives as a saved object (type security-ai-prompt) installed by the security_ai_prompts Fleet package. The local file in elastic_assistant is only a fallback when no saved object exists.

This POC keeps the packaged prompt untouched and adds a second variant resolved purely from local code, plus a request-time switch so the route can serve either one.

Why two prompt IDs instead of editing the original. Renaming the existing constant would break the local fallback for the existing saved object (which still has promptId: entityDetailsHighlights). Adding a second prompt ID with no matching saved object means the local-code text always wins for that ID, and the packaged behaviour stays identical for unchanged callers.

How getPrompt resolves a variant

The route passes a promptId to @kbn/security-ai-prompts, which checks saved objects first and only falls back to localPrompts when nothing matches.

  1. 1 Route receives promptVariant: 'default' | 'new' (optional). Default is default.
  2. 2 Route picks promptDictionary.entityDetailsHighlights for default, or promptDictionary.entityDetailsHighlightsNew for new.
  3. 3 getPrompt queries the SO index for security-ai-prompt rows matching that promptId + promptGroupId.
  4. 4 For entityDetailsHighlights, the packaged SO matches and is returned (unchanged behaviour).
  5. 5 For entityDetailsHighlightsNew, no SO exists, so localPrompts resolves it from ENTITY_DETAILS_HIGHLIGHTS_PROMPT_NEW.

Source: x-pack/solutions/security/packages/security-ai-prompts/src/get_prompt.ts (findPromptEntry).

Route handler — variant selection

const promptVariant = request.body.promptVariant ?? 'default';
const resolvedPromptId =
  promptVariant === 'new'
    ? promptDictionary.entityDetailsHighlightsNew
    : promptDictionary.entityDetailsHighlights;

const prompt = await getPrompt({
  getInferenceConnectorById: (id) => inference.getConnectorById(id, request),
  connectorId,
  promptId: resolvedPromptId,
  promptGroupId: promptGroupId.aiForEntityAnalytics,
  savedObjectsClient: soClient,
});

Schema change

The new field is optional, so existing callers (Kibana UI, tests, generated API clients) keep working without changes.

promptVariant:
  type: string
  enum:
    - default
    - new
  description: >-
    Dev-only: selects which prompt variant the server resolves for entity
    highlights. `default` (or omitted) uses the packaged saved-object prompt
    (`entityDetailsHighlights`). `new` uses the candidate replacement prompt
    (`entityDetailsHighlightsNew`) which is only resolved via the local
    fallback. Not intended for production callers.

Schema file: security_solution/common/api/entity_analytics/entity_details/highlights.schema.yaml

UI toggle

The entity flyout accordion now exposes a compressed button group next to the existing demo controls. Selecting a variant updates local state; the next time the analyst clicks Generate, the hook posts that variant to the route.

Docs generator changes

The entity-summary-analysis command measures storage impact of summaries persisted to the entity store. It does not call the LLM; it POSTs a placeholder body whose shape matches a real highlights output.

The script now carries two named placeholder bodies:

  • OLD_PROMPT_HIGHLIGHTS and OLD_PROMPT_RECOMMENDED_ACTIONS — sized like a typical legacy output.
  • NEW_PROMPT_HIGHLIGHTS and NEW_PROMPT_RECOMMENDED_ACTIONS — placeholder seeded with realistic structure for the new prompt; replace with a real captured output before running the comparison for keeps.

Replace the placeholders with real LLM output. The storage delta is only meaningful once both constants reflect actual output from each prompt variant. Generate two summaries via the entity flyout (toggle on Old prompt, then New prompt), copy the rendered cards into the script, then run --compare-prompts.

Flag matrix

Old and new flags on the entity-summary-analysis command
Flag Status Effect
--space Existing Kibana space to target.
--sample-pct / --sample-count Existing Single-run sampling controls.
--measure-existing Existing Skip persist; measure summaries already in the store.
--run-all Existing Full adoption sweep (5%, 10%, 25%, 50%, 75%, 100%).
--prompt-variant old|new New Pick which placeholder body to POST. Defaults to old.
--compare-prompts New Runs the full sweep twice (once per variant) against the same entity set and prints a side-by-side delta table.

What the comparison report shows

  • Per-sample blob size, doc-Δ, index-Δ, and list-API-Δ for both variants.
  • Headline blob change (absolute and percentage) from the largest sample.
  • Projected storage delta in MB at 5k / 10k / 25k / 50k entities and 10 / 50 / 100% adoption — i.e. how much more or less the new prompt would cost at scale.

How to use it locally

Quick visual check in the UI

  1. 1 Open an entity flyout and expand the Entity summary accordion.
  2. 2 Toggle between Old prompt and New prompt in the header controls.
  3. 3 Click Generate once per variant. The body of the response will reflect whichever prompt the server resolved.
  4. 4 Optionally inspect the prompt field in the network response to confirm which text the server sent to the model.

Storage impact comparison

# Old prompt only (default behaviour, unchanged)
yarn start entity-summary-analysis --run-all

# Single run, new prompt placeholder
yarn start entity-summary-analysis --run-all --prompt-variant new

# Full side-by-side: two passes, delta table at the end
yarn start entity-summary-analysis --compare-prompts

Run from the security-documents-generator repo.

POC safety

  • The promptVariant field is optional with a server-side default of default. Existing callers, tests, and the generated API client all continue to work without modification.
  • No saved object exists for entityDetailsHighlightsNew, so the new prompt is only reachable from code on this branch. Even if you later install or upgrade the security_ai_prompts Fleet package, it cannot collide.
  • The existing entityDetailsHighlights SO and its packaged text are untouched; uninstalling the package still uses the original ENTITY_DETAILS_HIGHLIGHTS_PROMPT as the local fallback.
  • The docs generator changes are additive. Without --prompt-variant or --compare-prompts, the command behaves exactly as it did before.

Cleanup before merging

This branch is a POC. Anything that does not need to ship should be reverted or feature-flagged.

  • Decide which prompt is the actual replacement. If ENTITY_DETAILS_HIGHLIGHTS_PROMPT_NEW wins, move its text into ENTITY_DETAILS_HIGHLIGHTS_PROMPT and delete the _NEW constant, the entityDetailsHighlightsNew dictionary entry, and the second localPrompts entry.
  • Remove the promptVariant field from the schema and route, regenerate highlights.gen.ts, and drop the parameter from the API client and hook.
  • Remove the prompt-variant button group from entity_highlights.tsx and the related state and toggle config.
  • Once a real new packaged prompt is decided, follow the steps in security-ai-prompts/README.md to regenerate the JSON, bump the security_ai_prompts integration version, and ship the updated package.
  • In security-documents-generator, decide whether to keep the comparison flag long-term. If yes, prune the placeholder constants to one canonical body (or keep both for legacy comparisons but mark them clearly).
