Entity store v2: overview, maintainers, and entity lifecycle

A walkthrough of how entity store v2 ingests, stores, and enriches entities — from raw security logs in the source indices to a fully decorated record in .entities.v2.latest.security_* — and how the five entity maintainers feed into that lifecycle.

Overview

TL;DR

One namespace, one latest index, one document per entity. Everything else in entity store v2 either feeds that document or enriches it.

Entity store v2 is a per-namespace store of Elastic entities — users, hosts, services, and generic entities — built from raw security logs. Each entity lives as a single document in the latest index, keyed by a hashed Elastic UID (EUID). Background tasks pull entities out of source indices, merge them with the existing record, and a set of entity maintainers enrich the record over time (risk score, watchlists, relationships, resolution).

The mental model that makes everything else easy to read:

  • The latest index is the single canonical surface. Everything else either feeds it or enriches it.
  • Extraction tasks write the base record from log evidence.
  • Maintainers rewrite specific fields on a schedule — they do not own the record, only the fields they enrich.
  • History snapshots are point-in-time copies of latest, written to a separate dated index.

This document describes v2 only. The change of shape between v1 and v2 is the easiest way to see what v2 buys you.

Previous

Entity store v1

  • One index per entity type per namespace.
  • Risk scoring lived in a separate risk_engine:risk_scoring task.
  • Enrichment work was spread across ad-hoc tasks with no shared framework.

Current

Entity store v2

  • One latest index per namespace for all entity types.
  • Risk scoring is one of the registered maintainers — same framework as the others.
  • v1 artefacts are removed on first install by stopAndRemoveV1().

Source: entity_store/server/plugin.ts, entity_store/server/infra/remove_v1.ts, entity_store/common/domain/entity_index.ts

Take-away. The flyout never asks "which maintainer wrote which field" at read time. Every piece of context Sarah sees was written by a different system on a different schedule and quietly merged into the same record. The next sections walk through each of those systems in turn.

Architecture

TL;DR

Six domain clients, one purpose: keep one document per entity correct and current. Routes pick the right client per request.

The plugin registers a small set of domain clients at setup, each with a focused responsibility. A consuming plugin or HTTP route resolves one of these clients per request via the request handler context.

Orchestrator

AssetManagerClient

Owns install, start, stop, uninstall, and aggregate status. Removes v1 artefacts on first install and schedules every entity store task.

server/domain/asset_manager/asset_manager_client.ts

Extraction

LogsExtractionClient and CcsLogsExtractionClient

Run the ES|QL extraction pipeline that turns raw logs into entity documents. The CCS variant stages remote-cluster updates into a separate data stream; the local client folds them in on the next run.

server/domain/logs_extraction/logs_extraction_client.ts and ccs_logs_extraction_client.ts

Read & write

CRUDClient and EntityStoreCRUDClient

Direct read and write access to the latest index. Other plugins receive a narrowed surface (EntityStoreCRUDClient) via the start contract so they cannot create or delete records directly.

server/domain/crud/crud_client.ts, server/types.ts

Enrichment

EntityMaintainersClient

Holds the registry of maintainers and schedules them as Task Manager tasks. Mediates the lifecycle transitions (never_started → started → stopped).

server/domain/entity_maintainers/entity_maintainers_client.ts, server/tasks/entity_maintainers/

Backup

HistorySnapshotClient

Runs the periodic reindex from latest into a dated history index, then resets per-snapshot counters on latest.

server/domain/history_snapshot/history_snapshot_client.ts

Identity

ResolutionClient

Links and groups entities that resolve to the same real-world identity. Updates resolution relationship fields on the latest document.

server/domain/resolution/resolution_client.ts

Indices

TL;DR

Three indices and one mental rule: every name is namespace-scoped and produced by a helper. Never hardcode a name in your code.

Three indices and data streams matter, all scoped to a namespace. Names are produced by helpers in common/domain/entity_index.ts so callers never hardcode the patterns.

Anatomy of the latest index name

Every part of the latest index name encodes something specific. Once you can read the parts, you can read any of the v2 index names by analogy.

.entities. v2 . latest . security _ {ns} -00002
.entities. — fixed prefix for the entity store family of system indices.
v2 — schema version. v1 indices are removed on v2 install.
latest — role: latest, history, or updates.
security — solution that owns the data (currently always security).
{ns} — Kibana space / namespace. Each namespace is fully isolated.
-00002 — rollover suffix. Callers use the stable alias entities-latest-{ns}.
Entity store v2 indices and data streams in namespace {ns}.
Role Concrete pattern Alias Written by
Latest (canonical) .entities.v2.latest.security_{ns}-00002 entities-latest-{ns} Extraction, maintainers, CRUD routes
Updates (CCS staging) .entities.v2.updates.security_{ns} (data stream) entities-updates-{ns} CcsLogsExtractionClient
History snapshot .entities.v2.history.security_{ns}.{YYYY-MM-DD-HH} (no stable alias) HistorySnapshotClient

The latest index uses a default ingest pipeline named .entities.v2.latest.security_{ns}_ingest_pipeline that expands dotted fields. Templates are installed once per namespace by installIndicesAndDataStreams() in server/infra/elasticsearch/install_assets.ts.

Why a separate updates data stream. Cross-cluster extraction cannot LOOKUP JOIN remote indices in one query, so remote updates are staged locally first. The next local extraction reads the updates DS as just another source pattern and merges the rows into latest using the same ES|QL pipeline.

Install and engine status

TL;DR

One HTTP call installs everything. Five of the six install steps are about pointing recurring tasks at the right targets.

Install flow

1

Install route called

Caller hits POST /api/security/entity_store/install. Privileges are checked and the request is forwarded to AssetManagerClient.init().

2

Remove v1 artefacts

init() calls stopAndRemoveV1() so v1 schemas cannot conflict with v2.

3

Create ES assets

installSharedElasticsearchAssets() creates pipelines, component and index templates, the latest index, and the updates data stream.

4

Per-type engines

For each entity type, initEntity() writes the engine descriptor with status installing, then start() flips it to started and schedules the extract task.

5

Schedule maintainers

entityMaintainersClient.init(req) schedules every registered maintainer that does not yet have a task document.

6

Global tasks

The history snapshot task and the status report task are scheduled at the global level.

Source: server/routes/apis/install/index.ts, server/domain/asset_manager/asset_manager_client.ts

Status states

There are two layers of status. The aggregate status is what callers usually check; per-engine status is what determines extraction behaviour for a single entity type.

Aggregate

EntityStoreStatus

Values: not_installed, installing, running, stopped, error. Computed by AssetManagerClient.calculateEntityStoreStatus() from the set of per-engine statuses. Returned by GET /api/security/entity_store/status.

Per engine

EngineStatus

Values: installing, started, stopped, updating, error. Persisted on the entity-engine-descriptor-v2 saved object with id entity-engine-descriptor-v2-{entityType}-{ns}.

Global

entity-store-global-state

Stores cross-engine config such as logExtraction.additionalIndexPatterns and history snapshot settings. One per namespace: entity-store-global-state-{ns}.

Log extraction

TL;DR

Extraction owns the base record. Maintainers own the rest. If a field is missing, look at one or the other — not both.

Extraction is the path that creates and refreshes the base entity record from log evidence. It runs as a Task Manager task per entity type.

The task

Each entity type registers its own task type: entity_store:v2:extract_entity_task:user, ...:host, ...:service, ...:generic. The task body is implemented in server/tasks/extract_entity_task.ts and delegates to LogsExtractionClient.extractLogs().

The ES|QL pipeline

buildLogsExtractionEsqlQuery() in server/domain/logs_extraction/logs_extraction_query_builder.ts assembles a query that:

  1. Reads logs from Security Solution data view indices plus optional additionalIndexPatterns (alerts indices are filtered out).
  2. Aggregates rows into one row per entity using EUID grouping.
  3. LOOKUP JOINs the existing latest document so it can merge previous values for fields like entity.behaviors.*, entity.lifecycle.*, and known relationships.
  4. Emits the merged shape, ready to upsert into latest.

Ingest

The result is bulk-upserted by ingestEntities() in server/infra/elasticsearch/ingest.ts using doc_as_upsert, with the hashed EUID as the document ID — that is what makes the write idempotent.

// Simplified shape of one extraction run
await ingestEntities({
  esClient,
  targetIndex: getLatestEntitiesIndexName(namespace),
  esIdField: HASHED_ID_FIELD,
  rows: esqlRows,
});

Extraction owns the base fields on the record — identifiers, lifecycle, observed behaviours from raw logs. Maintainers own everything else. If a field is missing on the latest record, the most likely culprits are either an ES|QL projection issue in the extraction query or a missing source index in additionalIndexPatterns.

Maintainers

TL;DR

Five maintainers, five colours, five personalities. Each one writes a specific subtree of the latest document and nothing else.

A maintainer is a registered enrichment job. Each one targets a specific subset of fields on the latest document and runs on its own Task Manager schedule. The shared response type that the UI consumes lives in common/entity_maintainers.ts.

Framework

Maintainers are registered through registerEntityMaintainerTask() in server/tasks/entity_maintainers/index.ts, which adds them to the in-memory EntityMaintainersRegistry and registers a dedicated Task Manager task type: entity_store:v2:entity_maintainer_task:{id}. Per-namespace task ids follow {id}:{namespace}. Execution is centralised in executeMaintainerRun(), which short-circuits when the task status is stopped.

Each maintainer is a tiny background worker with one job and one personality. Give them names and you'll find it easier to remember which one wrote which field on the latest document.

Asset criticality is not a maintainer. It is owned by the Asset Criticality API and stored on the entity record as entity.attributes.asset.criticality. The risk-score maintainer reads it at runtime via getCriticalityModifier() when computing a final score.

Anatomy of a maintainer run

Every maintainer follows the same loop. Task Manager fires the trigger, the maintainer checks its own status, reads from a source, transforms, and writes back through the CRUD client. If you understand one, you understand all five.

Lifecycle states

The taskStatus field on EntityMaintainerResponseItem can be one of three values. Each transition has a clear owner.

never_started

No Task Manager document exists yet

The maintainer is registered in code but has not been scheduled. EntityMaintainersClient.getMaintainers() reports this default when taskManager.get() returns NotFound. The only way out of this state is an explicit init.

started

Task is scheduled and runs are executing

Set by scheduleEntityMaintainerTask() on first init, and by startEntityMaintainer() when transitioning back from stopped. executeMaintainerRun() proceeds normally unless a runtime license check fails.

stopped

Task exists but runs are skipped

Set by stopEntityMaintainer() via updateTaskStatus(..., STOPPED). The Task Manager document remains so the schedule is preserved, but executeMaintainerRun() returns early when it sees this status.

Routes that drive transitions

Maintainer management routes and the lifecycle effect they have.
Route Effect
GET /internal/security/entity_store/entity_maintainers Returns the full GetEntityMaintainersResponse built by toGetMaintainersResponseItem().
POST /internal/security/entity_store/entity_maintainers/init Schedules any maintainer currently in never_started.
PUT /internal/.../entity_maintainers/start/{id} Sets the named maintainer to started and bulk-enables the underlying task.
PUT /internal/.../entity_maintainers/stop/{id} Sets the named maintainer to stopped.
POST /internal/.../entity_maintainers/run/{id}?sync=true|false Forces an immediate run, optionally awaiting completion.

Entity lifecycle timeline

TL;DR

Read the timeline left to right. The colour tells you which maintainer fired each event. The annotated document at the bottom shows what each one wrote.

This is the path a single entity travels, from a single log line landing in a source index to a fully decorated record in the latest index. Times are illustrative — actual intervals depend on the entity type and the maintainers enabled.

Lifecycle at a glance

Source indices feed an extract task that owns the base entity record. Five maintainers enrich specific fields on the same latest document. A separate periodic task copies the latest record into a dated history index.

Example: a new host called web-prod-01

Imagine an EDR agent on web-prod-01 emits its first event into logs-endpoint.events.process-default at 10:00:00. The following timeline shows what the entity store does with it.

  • Source log
  • Extract task
  • Maintainer
  • History snapshot
  1. 1 10:00:00 — Raw event indexed. The endpoint agent writes a process event to logs-endpoint.events.process-default. No entity record exists yet.
  2. 2 10:05:00 — Host extract task fires. Task Manager runs entity_store:v2:extract_entity_task:host. runTask() hands off to LogsExtractionClient.extractLogs().
  3. 3 10:05:01 — ES|QL extraction builds a row. buildLogsExtractionEsqlQuery() emits one row keyed by the host EUID, with base identifiers and observed behaviours. The LOOKUP JOIN finds no existing latest doc, so previous-state merging is a no-op.
  4. 4 10:05:02 — Latest document created. ingestEntities() upserts the row into .entities.v2.latest.security_default-00002 with the hashed EUID as the doc id. The UI now sees a barebones host record.
  5. 5 10:30:00 — Communicates-with maintainer runs. entity_store:v2:entity_maintainer_task:communicates_with reads recent network logs, builds the host's entity.relationships.communicates_with list, and calls crudClient.bulkUpdateEntity().
  6. 6 11:00:00 — Risk score maintainer runs. entity_store:v2:entity_maintainer_task:risk-score reads alerts for this host, applies the asset criticality modifier (none yet) and any watchlist modifiers (none yet), and writes entity.risk.calculated_score_norm via persistRiskScoresToEntityStore().
  7. 7 11:10:00 — Watchlist maintainer runs. An analyst added web-prod-01 to a "production hosts" watchlist at 11:05. entitySourcesService.syncAllWatchlists() writes entity.attributes.watchlists on the latest doc.
  8. 8 12:00:00 — Risk score maintainer runs again. The watchlist now influences the score. The same flow as step 6 produces an updated entity.risk.calculated_score_norm.
  9. 9 Top of the hour — History snapshot. HistorySnapshotClient.runHistorySnapshot() reindexes latest into .entities.v2.history.security_default.{YYYY-MM-DD-HH} and resets per-snapshot counters on latest.
  10. 10 Next day — Accesses-frequently maintainer runs. entity_store:v2:entity_maintainer_task:accesses_frequently_and_infrequently rebuilds the access relationship lists from yesterday's auth logs. The same bulkUpdateEntity() call pattern as the other relationship maintainer.

What an analyst sees. The flyout reads entity-store-{ns}-latest through CRUDClient.searchLatestEntities(). There is no aggregation of multiple maintainers at read time — each maintainer has already written its own fields to the latest document, so the UI just renders the merged record.

Extract task

Owns entity.id, type, name, source, lifecycle.*, behaviors.*, attributes.managed, and all ECS sibling fields (host.*, event.*).

watchlist maintainer

Writes entity.attributes.watchlists on every entity that matches an active watchlist source.

Asset Criticality API

Not a maintainer. The analyst sets criticality; risk-score reads it at runtime via getCriticalityModifier().

risk-score maintainer

Writes entity.risk.* hourly. The second run picks up the watchlist membership added between runs 1 and 2.

communicates_with maintainer

Writes entity.relationships.communicates_with.ids from recent network logs.

accesses_frequently_and_infrequently

Daily. Writes entity.relationships.accesses_frequently.ids (and the matching infrequently list) from auth logs.

After that, the history snapshot reset clears entity.behaviors.rule_names and entity.behaviors.anomaly_job_ids on the latest doc so behaviour lists reflect the next window rather than growing without bound.

What if a maintainer is stopped

Suppose an operator stops the watchlist maintainer via PUT /internal/.../entity_maintainers/stop/watchlist. The Task Manager task continues to be scheduled, but executeMaintainerRun() returns early, so entity.attributes.watchlists is no longer refreshed. The risk score keeps running and continues to apply previously written watchlist memberships until the maintainer is started again and a fresh sync replaces them.

CRUD client

TL;DR

All reads and writes funnel through the CRUD client. Maintainers get a narrower variant so they can't create or delete — only update.

Consuming plugins read and write entity records through EntityStoreCRUDClient, obtained from the entity store start contract. The maintainer framework passes a narrower variant called EntityUpdateClient into setup and run callbacks so maintainers cannot accidentally create or delete entities.

Methods on the server-side CRUDClient. The exported EntityStoreCRUDClient omits createEntity; the maintainer-scoped EntityUpdateClient also omits deleteEntity.
Method Purpose
createEntity(type, doc) Create a record from scratch. Not exposed to other plugins.
updateEntity(type, doc, force, options?) Patch or upsert a single record.
bulkUpdateEntity({ objects, force }) Batched updates from maintainers and admin routes.
deleteEntity(id) Delete a record by hashed EUID. Not exposed to maintainers.
listEntities(params?) Page or cursor through latest.
searchLatestEntities(params) Higher-level search wrapper used by UI surfaces.

Example: a consuming plugin writes a custom attribute

// In another plugin's server setup
const crudClient = entityStore.createCRUDClient(esClient, namespace);

await crudClient.bulkUpdateEntity({
  objects: [
    {
      type: 'host',
      doc: {
        entity: {
          id: hostEuid,
          attributes: { my_plugin: { tag: 'gold' } },
        },
      },
    },
  ],
  force: true,
});

The same call shape is what every maintainer uses, with a narrower EntityUpdateClient reference. There is no separate write path for maintainers versus other plugins, which is what keeps the record's structure consistent.

Cheat sheet

TL;DR

Three indices, four routes, five maintainers, five lifecycle states. If you remember those numbers, you can hold the whole thing in your head.

Everything in this document, distilled to a single page you can keep open next to your terminal.

Indices

  • .entities.v2.latest.* — current state. Read this.
  • .entities.v2.history.* — periodic snapshots. Read for audit only.
  • .entities.v2.creation.* — append-only raw extracts.
  • Names come from EntityIndexNames helpers — never hardcode.

Routes

  • POST /api/security/entity_store/install
  • GET /api/security/entity_store/status
  • POST /api/security/entity_store/maintainers/{id}/start
  • POST /api/security/entity_store/maintainers/{id}/stop

Maintainers

  • risk-score — hourly
  • watchlist — every 10 min
  • automated-resolution — daily
  • accesses_frequently_and_infrequently — daily
  • communicates_with — every 30 min

State values

  • Install status: not_installed, installing, installed, error, uninstalling
  • Maintainer status: never_started, started, stopped
  • Stopped maintainers stay scheduled — their run gates on status, the task itself is unaffected.

When something is wrong

  • No entity record: check extract task — base data comes from extraction, not maintainers.
  • Risk score stale: check risk-score maintainer status + Task Manager log.
  • Watchlist tag missing: check the watchlist source is active and its query matches.
  • Stale relationships: check communicates_with / accesses_* task runs.
  • Flyout empty: run searchLatestEntities by id in dev tools — the API only returns what is on the latest doc.

Read the document, not the joins. Every analyst surface — flyout, list, alert enrichment — reads the latest index by id. If it isn't on the document, the UI can't show it. Push problems back to the maintainer or extract task that owns that field.
