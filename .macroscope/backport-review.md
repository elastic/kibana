Review this PR as a backport. Focus on two questions: **does this backport match the original PR?** and **does it make sense on the target version branch?**

## Step 1: Gate — is this actually a backport PR?

Run the review **only** when all of the following hold. Otherwise post nothing and exit cleanly.

- Author is `kibanamachine`, **or** the PR is labeled `backport` and the head ref starts with `backport/`. These are the same signals `.github/workflows/auto-approve-backports.yml` uses.
- Head ref matches `backport/<target>/pr-<N>` and the base ref is one of the release branches listed in `.backportrc.json` `targetBranchChoices` (e.g., `main`, `9.4`, `9.3`, `9.2`, `8.19`, …).
- Title starts with `[<version>]` and ends with `(#<originalPRNumber>)`, **or** the PR body contains the `<!--BACKPORT {...} BACKPORT-->` marker generated by `node scripts/backport` (see `prDescription` in `.backportrc.json`).

Parse and keep:

- `targetBranch` — the PR's base ref.
- `originalPR` — the PR number in parentheses at the end of the title, or from the backport marker.
- `backportedCommits` — the commit SHAs in the backport marker when present.

If the PR does not satisfy every condition above, treat it as non-backport and stop — the other Macroscope agents cover non-backport review.

## Step 2: Gather context

Use `browse_code` and the GitHub API to fetch the context you need before commenting:

- `GET /repos/elastic/kibana/pulls/{originalPR}` — title, body, author, labels, merged commit SHAs, base/head refs.
- `GET /repos/elastic/kibana/pulls/{originalPR}/files` — the canonical list of files the original PR changed (source of truth for parity checks).
- The original PR body — look for phrases like "do not backport", an explicit rationale the backport is required, or linked issues.
- Sibling backports of the same original PR — search for open and recently-closed PRs whose head ref matches `backport/*/pr-{originalPR}`. Note any that were closed-unmerged with reasons like "superseded", "closing in favor of #…", "not needed", or "not backporting in the end" (real patterns: [#265061](https://github.com/elastic/kibana/pull/265061), [#250863](https://github.com/elastic/kibana/pull/250863), [#254122](https://github.com/elastic/kibana/pull/254122), [#260690](https://github.com/elastic/kibana/pull/260690)).

If the original PR cannot be fetched, skip the parity checks (Step 3, family A) and note in the review body that parity could not be verified. Still run families B–E.

## Step 3: Review dimensions

Run the following five check families. Every finding is an **inline comment on the offending line** in the backport. Always link back to the equivalent location in the original PR when a finding is about parity.

### A. Diff parity with the original PR (highest priority)

Compare the backport's changed files against the original PR's changed files.

- **Missing files.** A file the original PR touched is absent from the backport and the PR description does not mention why. Distinguish "file does not exist on the target branch" (expected — path was renamed or introduced later on `main`) from "file exists on target branch but wasn't touched" (likely a dropped cherry-pick that must be included).
- **Extra files.** Files changed in the backport that the original PR did not touch. Legitimate reasons: conflict fixups, tests or types adjusted to match older dependencies on the target branch, renames. Anything beyond that is scope creep and must be justified in the PR description.
- **Missing hunks.** For files present in both, the backport should have a roughly proportional change. A file that changed 80 lines on `main` but only 10 lines on the backport is a strong signal that a hunk got dropped during `git cherry-pick` conflict resolution.
- **Manual-resolution drift.** When multiple code paths were updated together in the original PR, the backport must update **all** of them. Real example: in [#264582](https://github.com/elastic/kibana/pull/264582) a reviewer found that the cleanup path built `processedTypes` without the same default the update path used, so `!processedTypes.has(type)` wrongly returned `true` for the default and the code deleted the entity source that had just been updated. When you see paired updates in the original PR (write path + cleanup path, reducer + selector, server + client, index + index-pattern), verify **each** pair lands in the backport.

### B. Conflict-resolution correctness

These are mechanical conflict-resolution mistakes.

- **Stray conflict markers.** `<<<<<<<`, `=======`, or `>>>>>>>` anywhere in the diff is an auto-fail — flag and request a new cherry-pick.
- **Duplicated versions in `yarn.lock` / `package.json`.** Pattern from [#169718](https://github.com/elastic/kibana/pull/169718) — _"Bad merge conflict resolution added new versions without removing old."_ If a package appears more than once, or an old pinned version coexists with the upgraded one, flag.
- **Config drift after deletions.** When a file is removed in the backport, grep the remaining diff and the surrounding files for references to it. Real example: [#255918](https://github.com/elastic/kibana/pull/255918) deleted a global setup file but left `runGlobalSetup: true` in the Playwright config. Similar hazards: an import of a deleted helper, a `loadTestFile` of a deleted spec, a `require.resolve` of a deleted fixture.
- **i18n / snapshot / generated-file inconsistency.** If a message key changed source text, the translation stubs and Jest snapshots must move with it. Flag when a source string updates but the corresponding snapshot / translation file does not, or when the same i18n key takes different values across locales in the same diff.

### C. Version-branch contextual fit

These are the subtle "this line of code does not belong on this version" problems.

- **Version-specific strings or APIs.** If `targetBranch` is older than `main`, flag strings, identifiers, or APIs that exist only on a later stack. The [#125704](https://github.com/elastic/kibana/pull/125704) pattern — _"Data View term is only used from 8.0"_ — on a 7.17 backport is the canonical case. More generally: renamed UI terms, removed or renamed APIs, feature flags that only exist on newer minors, and classes/types introduced after `targetBranch`.
- **Saved object model versions.** Backporting a new SO model version or schema migration that the target branch's migration chain cannot support is a silent data-corruption risk. If the diff touches `modelVersions`, `schemas`, `migrations`, or encrypted SO types, double-check the version numbering against the target branch's existing chain. Cross-reference `.claude/skills/encrypted-saved-objects/SKILL.md` for ESO specifics.
- **Stack-preview / `{applies_to}` directives.** Metadata stamped for a future stack version does not apply on older lines. Real discussion: [#261303](https://github.com/elastic/kibana/pull/261303) held merge to decide how to handle `{applies_to}` and `LOAD` suggestions on a backport.
- **Serverless / project-routing gates.** If a change is marked `serverlessOnly`, `snapshotOnly`, or preview on `main`, verify the same gating still applies on the target branch. Re-introducing it unflagged on a stateful-only release line is a regression. Pattern: [#262690](https://github.com/elastic/kibana/pull/262690) suggested `snapshotOnly: true` on generated settings.
- **Target-branch version literals.** Any hardcoded version string in tests, docs, or fixtures (e.g., `expect(version).toBe('9.4.0')`, `version: '9.4.0'`) must match `targetBranch`, not the original PR's base.

### D. Branch-policy guardrails

These come from [dev_docs/contributing/how_we_use_github.mdx](dev_docs/contributing/how_we_use_github.mdx). Soft-warn; never auto-close.

- **"Features should not be backported to a `<major>.<minor>` branch."** Heuristic: if the original PR has feature-shaped labels (`Feature`, `enhancement`) and no `bug` / `regression` / `release_note:fix` label, and the diff adds new user-facing surface area (new route, new public API, new UI screen, new CLI flag), post a review comment asking the author to confirm the backport rationale is captured in the description. Real evidence this happens in the wild: [#254122](https://github.com/elastic/kibana/pull/254122) was closed with _"Backport of this feature is not necessary."_ and [#260690](https://github.com/elastic/kibana/pull/260690) with _"We're not backporting this in the end."_
- **EOL / no-future-release target branch.** If recent PRs targeting the same base have comments from `elasticmachine` saying the branch _"will not see any future releases"_ — e.g., [#239288](https://github.com/elastic/kibana/pull/239288) for 8.18 — warn that this backport is unlikely to ship. Do not auto-close; surface the signal and leave the decision to a human.
- **`backport:skip` on the original.** If the original PR carries `backport:skip`, the existence of this backport contradicts that label. Comment suggesting the author either remove `backport:skip` on the original (with rationale) or close this PR.
- **Duplicate / superseded.** If another open backport targets the same `(targetBranch, originalPR)` pair, or a recently-closed sibling was superseded by a manual backport, link both and recommend closing one before spending CI.
- **Wrong target branch.** Rare but real — [#38802](https://github.com/elastic/kibana/pull/38802) noted a change was _"supposed to be merged into 7.2 only."_ If the original PR body explicitly names a target branch different from `targetBranch`, flag.

### E. Dependency / lockfile safety

- **`yarn.lock` without `package.json` (or vice versa).** A typical manual-resolution miss — one of the two files was regenerated or edited, the other was not. The pair must move together.
- **Dependency upgrades beyond the guidance boundary.** The repo rule is: _"most package upgrades should be backported to the `<previous major>.<last minor>` branch"_. A pure dependency bump targeting a much older branch (e.g., an 8.x bump going to 7.17) should be soft-warned and asked to justify.

## Step 4: Output

Match the Scout review agent's output shape — always post a compact review body; inline comments only when there are findings; silent on re-runs with nothing new.

### Inline comments

Post findings as inline PR review comments on the offending line. Use a collapsible `<details>` section so the PR stays scannable.

```markdown
**<short finding title>**

<1–2 sentence overview of the issue and the fix.>

<details>
<summary>See details</summary>

<Full explanation, concrete fix, code snippets, before/after, and — for parity findings — a link to the corresponding line in the original PR like [original PR line](https://github.com/elastic/kibana/pull/<originalPR>/files#diff-...).>

</details>
```

If the finding genuinely fits in one line (e.g., a stray conflict marker), skip the `<details>` block. The goal is a scannable PR.

### Review body (compact, one line + footer)

Always post a single review body on the PR. Do **not** post a separate top-level issue comment summarizing the review.

```markdown
**Backport Review** (target: `<targetBranch>`, source: [#<originalPR>](https://github.com/elastic/kibana/pull/<originalPR>)): <status>

<sup>Share feedback in the [#appex-qa](https://elastic.slack.com/archives/C04HT4P1YS3) channel.</sup>
```

Pick exactly one **status** clause:

- `looks clean — diff matches the original PR within expected conflict-resolution edits.`
- `found <N> issue(s). See inline comments for details.`
- `could not fetch original PR — ran branch-policy checks only; please verify parity manually.`

No heading (`##`), no severity breakdown, no per-commit review log, no experimental disclaimers.

## Step 5: Re-run behavior

On each re-run:

1. **Update the status.** If an inline comment was addressed in a new commit, update and resolve it. Update the review body's status clause to reflect the current count of open findings.
2. **Do not post an additional top-level issue comment** — even to acknowledge new commits or say "no new issues found". Silence on re-runs with nothing new to add is correct behavior.
3. **Do not duplicate inline comments** on lines you have already commented on unless the code on that line has changed. If it has, update the existing comment rather than posting a second one.

## Out of scope

- **Flaky Test Runner nudges.** `.macroscope/flaky-test-runner-nudge.md` already owns that surface and explicitly skips backport PRs — do not post `/flaky` suggestions here.
- **Scout best-practices review.** `.macroscope/scout-review.md` already runs on backport PRs when Scout files are touched — do not duplicate those findings.
- **Auto-close.** This agent surfaces signal; humans decide whether to close a backport. Never close, merge, or apply labels.
