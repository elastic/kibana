### Remarks discovered during A1

- The earlier discovery report claimed the type-check base sets `moduleResolution: "bundler"`. **Wrong.** Both bases currently use `moduleResolution: "node"` (verified at `tsconfig.base.type_check.json:8348`). That means A3 (`node` ŌåÆ `bundler`) is a real flip on both bases, not a unification.
- The earlier report claimed `tsconfig.base.type_check.json` is hand-maintained and the two bases need manual sync. **Partially wrong.** It IS auto-generated by `createTypeCheckConfigs` in `@kbn/ts-type-check-cli/execute_type_check_validation.ts` whenever a project whose `extends` chain resolves to `tsconfig.base.json` is built. The dual-base "maintenance trap" only exists if someone hand-edits the generated file (or if a full type-check has not run since the IDE base changed). A2 might be unnecessary; reframe it as "verify the generator is the single source of truth, and document it".
- Bootstrap (`yarn kbn bootstrap`) ran cleanly. `config-paths.json` already lists `@kbn/apm-common`; an earlier `node scripts/type_check` failure (`@kbn/apm-common does not point to another TS project`) **pre-existed on the branch** (reproduced on `git stash`) and resolved itself after a fresh bootstrap rebuilt the project graph. Likely stale yarn link state from prior branch switches.

### A3. Switch `moduleResolution` `"node"` ŌåÆ `"bundler"` ŌĆö DEFERRED

- Status: probed and reverted. Splitting into multiple fix PRs.
- Paired change required: `module: "commonjs"` ŌåÆ `"preserve"` (or `"esnext"`). `"bundler"` only works with `module: preserve|es2015+`. Verified `module: "preserve"` works for the base.
- Probe results on `src/core/packages/plugins/server-internal` (large sweep): **121 errors** across the dependency closure. Browser-internal: 0 errors. Small packages (kbn-ts-projects): 0 errors.
- Error breakdown (top categories from probe):
  - 43 ├Ś TS2305 (no exported member) ŌĆö mostly `@elastic/opentelemetry-node/sdk` subpath types not exposing the expected members under bundler resolution. The ambient typings in `typings/@elastic/opentelemetry-node/sdk` expose `tracing`/`core`/`api`/`node`/`resources` but only get picked up by `node10`-style classic resolution. Bundler resolution looks at the package's real `exports` map.
  - 37 ├Ś TS2322 / 13 ├Ś TS2344 / 3 ├Ś TS2339 ŌĆö downstream type incompatibility cascading from the above resolution change.
  - 6 ├Ś TS1192 ŌĆö `Module '@elastic/elasticsearch/lib/api/types' has no default export`. Source uses `import api from '@elastic/elasticsearch/lib/api/types'`; under bundler this must be `import * as api from ...`.
  - 6 ├Ś TS2578 ŌĆö unused `@ts-expect-error` (some prior workarounds become unnecessary; minor cleanup win).
  - 1 ├Ś TS2307 ŌĆö deep `@opentelemetry/api/build/src/metrics/Metric` import is not part of any public `exports`; correct fix is to import from the package root.
- Sequencing recommendation (each becomes its own small PR, all on `module: "node"`):
  1. Fix `@elastic/elasticsearch` deep imports (replace `import x from '@elastic/elasticsearch/lib/api/types'` with namespace import).
  2. Fix the `@opentelemetry/api` deep import.
  3. Either re-shape `typings/@elastic/opentelemetry-node/sdk` as a proper module with `exports`, or add proper re-exports to a real `@elastic/opentelemetry-node` types shim package.
  4. Audit and remove the unused `@ts-expect-error` lines surfaced (TS2578).
  5. Then flip the base flag in a final small PR (the actual one-line change is `"moduleResolution": "bundler"` + `"module": "preserve"`).
- This deferral is consistent with the plan's stated caveat: "Likely the largest individual PR; can be split per solution dir if needed." Splitting now is cheaper than reverting later.
- A3 is the **gating blocker** for TS 7.0 (which removes `moduleResolution: "node"` entirely). Phase C (TS 6.0 bump) can proceed with `ignoreDeprecations: "6.0"` if A3 is not finished by then.

### A4. Audit `@ts-expect-error upgrade typescript v5.9.3` comments ŌĆö DEFERRED to C2

- Status: surveyed, not modified.
- Surface: ~hundreds of occurrences across plugins (Fleet alone has 36 in `epm/archive/parse.ts`). A blanket cleanup PR is unmaintainable and review-hostile.
- Better strategy: keep them in place. When Phase C bumps TS 5.9.3 ŌåÆ 6.0.3, `@ts-expect-error` becomes a self-policing canary ŌĆö every now-unused suppression fails as TS2578 and gets deleted in the same C2 PR as a natural fallout. Anything still needed gets retagged to `upgrade typescript v6.0.3`.
- A4 is therefore folded into C2 rather than executed pre-bump.

### A5. Verify `@typescript-eslint` TS 6 compatibility

- Status: research done; bump deferred until just before Phase C (to minimize lockfile churn).
- Installed: `@typescript-eslint/*@8.46.3`. Declares `SUPPORTED_TYPESCRIPT_VERSIONS = ">=4.8.4 <6.0.0"` (verified in `node_modules/@typescript-eslint/typescript-estree/dist/parseSettings/warnAboutTSVersion.d.ts`). On TS 6.0.3 it would only print a runtime warning, not hard-fail ŌĆö but we want the supported range.
- Target: `@typescript-eslint/*@8.60.0` (latest as of fetch). Its `package.json` declares `peerDependencies.typescript: ">=4.8.4 <6.1.0"`. Covers TS 6.0.3.
- Bump strategy: do this as the first sub-PR inside Phase C, immediately before the `typescript` bump. Single yarn add of the 4 pinned packages (`@typescript-eslint/eslint-plugin`, `parser`, `typescript-estree`, `**/@typescript-eslint/utils` resolution). Renovate's `eslint` group will pick it up on its own cadence; do this manually to keep Phase C self-contained.
