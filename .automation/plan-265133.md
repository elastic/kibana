# Implementation Plan: 265133 — [Discover] Rename `dataStream` field to `sourceName` across the metrics types

## Issue Summary

The `dataStream` / `data_stream` field on the metrics types in `@kbn/unified-chart-section-viewer` is a misnomer: the `METRICS_INFO` ES|QL response returns whatever index or data-stream name the source query referenced, with no guarantee it is actually a data stream. The naming pushes a kind-assumption into the types and forces every consumer (parser, flyout, RED charts, grid) to read `dataStream` even when the value is conceptually agnostic. Rename to a kind-agnostic name — `sourceName` (singular) and `sourceNames` (plural) — across the package's parsed/in-memory types, all internal consumers (parser, ESQL query builder, flyout overview, metrics grid, unique-key util, telemetry accumulator), all tests, and the i18n placeholder in `metrics_experience_grid_content.tsx`.

The wire-format `data_stream` field name on `MetricsESQLResponse` (which mirrors the ES|QL column) is in scope but treated more conservatively: it is renamed only at the TypeScript boundary while a parser-level fallback continues to accept `data_stream` from the raw response payload (see "Open Questions" below).

## Acceptance Criteria

- [ ] `dataStream` is renamed to a kind-agnostic field (`sourceName`) on `ParsedMetricItem`.
- [ ] `dataStreams` is renamed to `sourceNames` on the internal `Metric` type.
- [ ] `multi_value_counts.data_streams` is renamed to `multi_value_counts.source_names` on `MetricsTelemetry` and the EBT schema is updated to match.
- [ ] `MetricsESQLResponse.data_stream` is renamed to `MetricsESQLResponse.source_name` at the TypeScript level; parsing still accepts the underlying ES|QL response column name (`data_stream`) so the change is invisible to ES|QL responses.
- [ ] All internal consumers (parser, ESQL query builder, flyout overview, metrics grid, unique-key util, telemetry accumulator, hooks) are updated to use the new names.
- [ ] All unit tests, snapshots, and mocks are updated to use the new names; tests still pass.
- [ ] The user-facing duplicate-metric description i18n string uses a kind-agnostic placeholder (`{sourceName}` instead of `{dataStream}` in the template values) — the visible label may stay "data streams" / "Data stream" since that is still semantically accurate when the kind is in fact a data stream.
- [ ] No semantic regression: existing behavior in the metrics grid, flyout, and trace RED charts remains identical.
- [ ] The new field name does not assume the underlying source is a data stream.

## Implementation Steps

### Step 1: Decide on `MetricsESQLResponse.data_stream` rename strategy
**Goal**: Confirm the renaming approach for the wire-shaped type. The ES|QL response column is literally named `data_stream`. Two options:

- **(A) Rename at TS boundary only, keep parser fallback.** `MetricsESQLResponse.source_name` (TS) but the parser reads `metric.source_name ?? metric.data_stream` so a still-on-the-wire `data_stream` column does not break parsing. Safer; aligns the TS surface with the rest of the rename; the runtime parser remains tolerant during the transition.
- **(B) Rename at TS boundary only, drop fallback, trust ES|QL response coerces by index.** The `parseMetricsWithTelemetry` function reads the raw response objects. If ES|QL still returns a column literally named `data_stream`, accessing `.source_name` returns `undefined` and `toArray(undefined)` returns `[]`, breaking the parser silently. **DO NOT pick this option** without first confirming the ES|QL column shape.
- **(C) Do not rename `MetricsESQLResponse` at all** — only rename downstream `ParsedMetricItem` and friends. This leaves the misnomer on the response-shaped type but is the lowest-risk option.

**Default**: Option (A). Leaves the public type semantically clean while protecting against ES|QL response shape coupling. Flag this in the PR for reviewer sign-off.

**Action**: Add an inline note in the parser explaining why the fallback exists ("ES|QL `METRICS_INFO` returns a column literally named `data_stream`; the TS type exposes it as `source_name`. Accept either at parse time so a future ES|QL column rename can land before or after this Kibana rename.").

**Notes**: This is the only "judgment call" in the plan. If the implementing agent prefers Option (C) at review time, the scope shrinks: skip the `MetricsESQLResponse` rename, skip the parser fallback, only do the in-memory `ParsedMetricItem` / `MetricsTelemetry` / `Metric` renames.

## Potential Risks & Considerations

- **EBT schema rename is breaking for downstream analytics.** Any dashboard or query that filters on `multi_value_counts.data_streams` will silently miss data after deploy. **Mitigation**: Call this out explicitly in the PR description; tag the obs-exploration team for sign-off; consider whether to publish both fields temporarily (emit `data_streams` and `source_names` with the same value) for one minor release before removing `data_streams`. Default plan does not dual-emit — flag this risk in the PR.
- **ES|QL `data_stream` column coupling.** The parser depends on the literal ES|QL column name returned by `METRICS_INFO`. Option A (Step 1) keeps a fallback to mitigate, but if ES|QL itself ever renames its column, both branches would still break. Adding a guard test for the fallback locks the contract.
- **`UnifiedMetricsGridRestorableState` cross-version compatibility.** Restorable state may serialize the unique key (`getMetricUniqueKey`). Since the key string format is preserved (only the field name read from `ParsedMetricItem` changed), keys serialized in pre-rename sessions remain valid against the post-rename code. Verify by inspecting `src/restorable_state.ts` to confirm no field name is itself serialized — only computed values. (This file is not currently flagged for changes but should be eyeballed during Step 2.)
- **Test name churn.** Many test descriptions include the word `dataStream`. Renaming them all is low-value churn; renaming only those that describe field semantics (vs source-kind labels) keeps the diff focused.
